# データ移行設計 作成ルール

Data Migration Design Documentation Rules

本ドキュメントは、`dmd-index`（データ移行設計の入口・共通SSOT）を一貫した粒度で作成するためのルールを定義する。

## 1. 全体方針

- `dmd-index` はデータ移行設計の入口（共通SSOT）として、抽出/変換/除外/冪等性/検証の共通方針を定義する。
- 個別データ領域の詳細は `dmd-<term>` に委譲し、`dmd-index` では共通判断軸と導線を定義する。
- 曖昧表現を避け、再実行条件と合否判定が可能な記述にする。

## 2. 位置づけと用語定義（必要に応じて）

- `mip-index`: 移行計画・成功条件・停止許容時間の合意
- `dmd-index`: データ移行の共通設計（本書）
- `dmd-<term>`: 対象別の詳細設計（抽出条件/マッピング/手順/検証）
- `mtp-index` / `mtp-<term>`: リハーサル計画と証跡設計
- `cop-index` / `cop-<term>`: 当日実行の詳細手順（Runbook）

## 3. ファイル命名・ID規則

- `id` は `dmd-index` を使用する。
- ファイル名は `dmd-index-データ移行設計.md` など、プロジェクト内で一意になるように命名する。

## 5. 本文構成（標準テンプレ）

| 番号 | 見出し                                            | 必須 |
| ---- | ------------------------------------------------- | ---- |
| 1    | 概要（index）                                     | ○    |
| 2    | 共通方針（抽出/変換/除外/冪等性/ログ）            | ○    |
| 3    | run_id 規約（命名/採番/記録/参照方法）            | ○    |
| 4    | 検証の共通方針（種別/基準/証跡/保管）             | ○    |
| 5    | 対象一覧（スコープ/非対象/担当/対応ドキュメント） | ○    |
| 6    | 関連ドキュメント（必須）                          | ○    |

## 6. 記述ガイド

### 6.1. 概要（index）

- データ移行設計の目的（再現性・品質担保）と範囲を 1〜3 行で示す。
- 本書が入口（SSOT）であり、詳細は `dmd-<term>` にあることを明記する。

### 6.2. 共通方針（抽出/変換/除外/冪等性/ログ）

- 抽出（フル/増分）の採用方針と増分基準を定義する。
- 変換/除外の原則（再実行で同一結果）と根拠リンクの方針を定義する。
- 冪等性とログ要件（最低限残す項目）を定義する。

### 6.3. run_id 規約（命名/採番/記録/参照方法）

- 命名規則（例: `DMD-YYYYMMDD-<term>-NN`）と採番方法を定義する。
- run_id をログ/投入データ/照合レポートに紐づける。

### 6.4. 検証の共通方針（種別/基準/証跡/保管）

- 最低限の検証種別（件数/サンプル/重要値）を定義する。
- 合格基準（差分0 / 許容差ε）の記述方法を定義する。
- 証跡の保管場所と命名規約を定義する。

### 6.5. 対象一覧（スコープ/非対象/担当/対応ドキュメント）

- 対象/非対象/担当/対応 `dmd-<term>` を一覧化する。
- `mip-index` の移行対象と整合し、導線を明記する。

### 6.6. 関連ドキュメント（必須）

- 各 `dmd-<term>` への導線を必ず記載する。
- 分冊が無い場合でも `（本書のみ）` を明記する。

## 7. 禁止事項

- `dmd-<term>` に記載すべき個別の変換仕様や実行手順を `dmd-index` に重複記載しない。
- 対象一覧に対象/非対象や対応DMDの導線を欠落させない。
- run_id 規約や検証共通方針を未定義のままにしない。
- 実装依存の長大なSQLやコードを本文に記載しない。
