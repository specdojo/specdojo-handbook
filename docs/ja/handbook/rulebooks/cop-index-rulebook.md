# カットオーバー計画（本番切替手順） 作成ルール

Cutover Plan Index Documentation Rules

本ドキュメントは、カットオーバー計画（本番切替手順）の全体SSOTである `cop-index` を統一形式で記述するためのルールです。
当日の切替を分単位で実行できる状態にし、責任分界とロールバックを明確にするための内容を定義します。

## 1. 全体方針

`cop-index` は、当日の切替を分単位で実行可能にし、判断と責任の所在を明確にするためのSSOTとする。

- 当日タイムラインと作業手順を時系列で定義する。
- 切替判定ポイント（GO/NO-GO）を明記する。
- ロールバック手順を具体的かつ実行可能に記述する。
- 担当者、連絡系統、エスカレーションを明確にする。
- 周知（ユーザ通知、停止告知）の手順を必須で含める。
- 分冊（`cop-<term>`）がある場合でも、全体前提と判断体系は `cop-index` を正とする。

## 2. 位置づけと用語定義（必要に応じて）

`cop-index` と他ドキュメントの関係を示す。

- `mip-index`: 移行の合意、判断のSSOT（成功条件、カットオーバー骨格）。
- `dmd-index`, `dmd-<term>`: データ移行の詳細設計（マッピング、変換、検証）。
- `mtp-index`, `mtp-<term>`: リハーサル計画、検証設計（観点、手順、証跡）。
- `cop-index`: 当日実行の詳細手順（Runbook）。
- `otp-index`, `otp-<term>`: 運用切替の詳細（監視設定、運用手順、窓口）。

必要に応じて用語を定義し、命名ゆれを防ぐ。

## 3. ファイル命名・ID規則

- `id` は `cop-index` とする。
- 推奨ファイル名は `cop-index` とする。
- プロジェクト内の実ファイル名は一意になるよう命名する。

## 4. 推奨 Frontmatter 項目

### 4.1. 設定内容

Frontmatter は共通スキーマに従う。

- 参照: `docs/shared/schemas/deliverable-frontmatter.schema.yaml`
- 参照: `meta-deliverable-metadata-rulebook.md`

| 項目       | 説明                             | 必須 |
| ---------- | -------------------------------- | ---- |
| id         | `cop-index`                      | ○    |
| type       | `migration`                      | ○    |
| title      | カットオーバー計画: 全体         | ○    |
| status     | `draft` / `ready` / `deprecated` | ○    |
| based_on   | 根拠となる仕様ID（配列）         | 任意 |
| supersedes | 置き換え関係（配列）             | 任意 |

### 4.2. 推奨ルール

- `based_on` には切替判断に直接利用した成果物のみを列挙する。
- `based_on` と `supersedes` は配列で記載する。

## 5. 本文構成（標準テンプレ）

`cop-index` は以下の見出し構成を順序固定で配置する。

| 番号 | 見出し                       | 必須 |
| ---- | ---------------------------- | ---- |
| 1    | 概要（index）                | ○    |
| 2    | 当日タイムライン             | ○    |
| 3    | 作業手順チェックリスト       | ○    |
| 4    | 担当者、連絡系統             | ○    |
| 5    | 実施条件（事前完了条件）     | ○    |
| 6    | 切替判定ポイント（GO/NO-GO） | ○    |
| 7    | ロールバック手順             | ○    |
| 8    | 障害時判断、エスカレーション | ○    |
| 9    | 周知（ユーザ通知、停止告知） | ○    |
| 10   | 関連ドキュメント             | ○    |

## 6. 記述ガイド

### 6.1. 概要（index）

- 切替対象と目的を1〜3行で示す。
- `mip-index` の骨格、判定点、ロールバック方針と整合させる。

### 6.2. 当日タイムライン

- 分単位で時刻を記述する（相対時刻を推奨）。
- 完了条件と判定点（GO1, GO2...）を明記する。

推奨フォーマット:

| 時刻（相対） | イベント | 完了条件 | 判定点 |
| ------------ | -------- | -------- | ------ |

### 6.3. 作業手順チェックリスト

- 作業を順序付きで列挙する。
- 実施者と証跡（ログ、チケット、チェック欄）を含める。

推奨フォーマット:

| 手順 | 作業内容 | 実施者 | 証跡 | 備考 |
| ---- | -------- | ------ | ---- | ---- |

### 6.4. 担当者、連絡系統

- 担当、責務、連絡経路、エスカレーション先を明記する。
- 個人名を出せない場合はロール名と窓口で表現する。

推奨フォーマット:

| 役割 | 担当 | 連絡先 | 備考 |
| ---- | ---- | ------ | ---- |

### 6.5. 実施条件（事前完了条件）

- 実施前に完了している条件を列挙する。
- 未達の場合の扱い（延期、代替手順、判断者）を明記する。

### 6.6. 切替判定ポイント（GO/NO-GO）

- 判定点、判断基準、判断者を列挙する。
- 失敗時の扱いは「ロールバック、延期、代替手順」のいずれかを明記する。
- `mip-index` の成功条件、GO/NO-GO条件と矛盾させない。

推奨フォーマット:

| 判定点 | 条件 | 判断者 | 失敗時の扱い（ロールバック、延期、代替手順） |
| ------ | ---- | ------ | -------------------------------------------- |

### 6.7. ロールバック手順

- 発動条件を明記する。
- 復旧方針（旧系復帰、接続戻し、投入データ無効化など）を明記する。
- 責務、証跡、完了条件の粒度で実行手順を書く。

### 6.8. 障害時判断、エスカレーション

- 障害時の判断フローと連絡経路を明記する。
- 重大度、SLO逸脱、復旧見込みなどの判断基準を明記する。

### 6.9. 周知（ユーザ通知、停止告知）

- 通知対象、通知手段、通知タイミングを明記する。
- 停止時間、影響範囲、問い合わせ先を必ず含める。
- 配信ログ、掲載URL、チケットなどの証跡を残す。

### 6.10. 関連ドキュメント

- 分冊がある場合は `cop-<term>` への導線を必ず記載する。
- 分冊がない場合でも `（本書のみ）` を明記する。

## 7. 禁止事項

| 禁止事項                                   | 理由                               |
| ------------------------------------------ | ---------------------------------- |
| 当日タイムラインが分単位で記載されていない | 実行性が担保できないため           |
| ロールバック条件や手順を省略する           | 失敗時の判断ができないため         |
| 担当者や連絡系統が不明確                   | 判断と責任が曖昧になるため         |
| 周知（ユーザ通知、停止告知）を未記載にする | 利用者混乱と事故につながるため     |
| `cop-index` と分冊の方針に矛盾を作る       | SSOTが崩れ、当日参照が割れるため   |
| 関連ドキュメントを省略する                 | 導線が失われ、参照先が分散するため |
