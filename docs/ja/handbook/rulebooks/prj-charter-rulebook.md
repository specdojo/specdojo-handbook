# プロジェクト憲章 作成ルール

Project Charter Documentation Rulebook

本ドキュメントは、プロジェクト憲章を統一形式で記述するためのルールです。
関係者が「誰が、何を、どの条件で承認したか」を監査可能な粒度で残すことを目的とします。

## 1. 全体方針

- 本ルールの対象は、プロジェクトの正式認可と権限委譲を示す「プロジェクト憲章」です。
- 目的は、開始可否の意思決定に必要な前提、責任、判定基準を明文化することです。
- プロジェクト憲章は「何を決める文書か」と「何を決めない文書か」を明示します。
- 背景・目的・必要性・期待効果の詳細本文は [prj-overview-rulebook.md](prj-overview-rulebook.md) を正とし、本書は認可判断に必要な要約と参照を記載します。
- 非機能要件、テスト戦略、運用準備の着手条件に触れますが、設計詳細や実装仕様は記載しません。
- 曖昧表現（例: 「適切に」「十分に」）は避け、判定可能な条件で記述します。

## 2. 位置づけと用語定義（必要に応じて）

### 2.1. 位置づけ（他ドキュメントとの関係）

プロジェクト憲章と関連ドキュメントの関係を示します。

```mermaid
flowchart LR
  POV["プロジェクト概要"]
  PCH["プロジェクト憲章"]
  PSC["プロジェクトスコープ"]
  PM["プロジェクト管理計画"]

  POV --> PCH --> PSC --> PM

  classDef target stroke-width:4px
  class PCH target
```

### 2.2. 用語定義（本ルール内）

| 用語       | 定義                                                                 |
| ---------- | -------------------------------------------------------------------- |
| スポンサー | プロジェクトの実施を正式に認可し、主要判断に責任を持つ役割           |
| 成功条件   | プロジェクト完了を判定するための基準（定量/定性）                    |
| 権限委譲   | プロジェクトマネージャーに付与する意思決定範囲と制約                 |
| 承認事項   | 誰が何を承認したかを示す記録対象                                     |
| 証跡       | 承認・判定・変更履歴を追跡できる記録（会議体、チケット、議事録など） |

## 3. ファイル命名・ID規則

### 3.1. 配置（推奨）

- `docs/ja/projects/<project-id>/010-プロジェクト定義/` 配下への配置を推奨します。
- 関連資料（承認議事録、スポンサー合意メモ等）は同ディレクトリ配下に配置し、本文から参照できる状態にします。
- ファイル名を日本語にする場合も、frontmatter の `id` で一意性を担保します。

### 3.2. ドキュメントID（推奨）

- 推奨: `<project-id>-prj-charter`
  - 例: `prj-0001-prj-charter`

### 3.3. ファイル名（推奨）

- 推奨: `prj-charter.md`
- 日本語ファイル名の場合: `プロジェクト憲章.md`

## 4. 推奨 Frontmatter 項目

### 4.1. 設定内容

- 参照スキーマ: [docs/shared/schemas/deliverable-frontmatter.schema.yaml](../../../../shared/schemas/deliverable-frontmatter.schema.yaml)
- メタ情報ルール: [meta-deliverable-metadata-rulebook.md](meta-deliverable-metadata-rulebook.md)

| 項目       | 説明                               | 必須 |
| ---------- | ---------------------------------- | ---- |
| id         | `<project-id>-prj-charter`         | ○    |
| type       | `project` 固定                     | ○    |
| status     | `draft` / `ready` / `deprecated`   | ○    |
| based_on   | 参照した方針書、契約条件、上位計画 | 任意 |
| supersedes | 置き換え対象の旧文書 ID            | 任意 |

### 4.2. 推奨ルール

- `based_on` は、承認判断の根拠として直接参照した資料のみを列挙します。
- 改訂時は `supersedes` に旧文書 ID を設定し、差し替え関係を追跡可能にします。
- H1 にはプロジェクト名を含め、Frontmatter には `title` を持たせません。

## 5. 本文構成（標準テンプレ）

プロジェクト憲章は以下の見出し構成を順序固定で配置します。

### 5.1. プロジェクト憲章（Project Charter）

| 番号 | 見出し                 | 必須 | 内容（要点）                                 |
| ---- | ---------------------- | ---- | -------------------------------------------- |
| 1    | 認可目的と成功判定対象 | ○    | 認可の対象、成功判定の対象、概要根拠への参照 |
| 2    | 認可対象範囲と委譲境界 | ○    | 承認対象範囲、権限委譲境界、対象外の明示     |
| 3    | 主要成果物と成功条件   | ○    | 成果物一覧、合格判定基準、測定方法           |
| 4    | 体制と権限委譲         | ○    | スポンサー、PM、意思決定者、権限境界         |
| 5    | 前提・制約・依存関係   | ○    | 予算、期限、外部依存、リスク前提             |
| 6    | 承認と証跡             | ○    | 承認者、承認日、承認経路、証跡リンク         |
| 7    | 未解決事項             | 任意 | 今後の意思決定が必要な論点、期限、担当       |

## 6. 記述ガイド

### 6.1. 共通

- 事実（観測値/制約）と判断（方針/選択理由）を分離して記述します。
- 章参照は章番号ではなく章タイトルで記述します。
- 未確定事項は `_UNDECIDED_:`、仮置き前提は `_ASSUMPTION_:` を用いて明示します。
- 背景や期待効果の詳細説明を本書に重複転記せず、[prj-overview-rulebook.md](prj-overview-rulebook.md) への参照で整合を保ちます。

### 6.2. 主要成果物と成功条件

- 成功条件は「条件」「判定基準」「測定タイミング」を最低 1 セット以上で記載します。
- 非機能観点（性能、可用性、セキュリティ、保守性）を最低 1 項目ずつ確認し、該当なしなら理由を記載します。
- テスト/受入の成立条件（例: 業務受入条件・システム受入条件への接続）を記載します。

推奨表（成功条件一覧）:

| ID  | 条件 | 判定基準 | 測定方法 | 判定時期 | 責任者 |
| --- | ---- | -------- | -------- | -------- | ------ |

### 6.3. 体制と権限委譲

- 承認権限（最終決裁）と実行権限（運営判断）を分けて記述します。
- 変更要求、予算超過、納期変更のエスカレーション経路を定義します。
- 役割ごとに責任範囲と不在時代理を明示します。

推奨表（権限マトリクス）:

| 項目 | 決裁者 | 実行責任者 | 協議先 | SLA/期限 |
| ---- | ------ | ---------- | ------ | -------- |

### 6.4. 承認と証跡

- 承認履歴は、日付、承認者、承認対象、証跡リンクを必須項目とします。
- 口頭合意のみで完了扱いにせず、会議体記録またはチケットを残します。
- 改版時は差分要約と再承認要否を明記します。

推奨表（承認履歴）:

| 版  | 承認日 | 承認者 | 承認対象 | 証跡リンク |
| --- | ------ | ------ | -------- | ---------- |

### 6.5. プロジェクト概要との分担

- 本書は「誰が、何を、どの条件で承認したか」を扱います。
- 背景、必要性、期待効果、前提条件の詳細は [prj-overview-rulebook.md](prj-overview-rulebook.md) を正とします。
- 本書に背景を記載する場合は要約のみに留め、根拠本文は概要側へリンクします。

## 7. 禁止事項

| 項目                                    | 理由                               |
| --------------------------------------- | ---------------------------------- |
| 設計詳細（DB定義、API項目詳細等）の記載 | 設計ドキュメントへ責務委譲するため |
| 判定不能な曖昧語のみで成功条件を定義    | 受入判定と監査ができないため       |
| 承認者未記載のまま正式認可と扱う        | 意思決定責任が不明確になるため     |
| 証跡リンクなしで承認済みと記載          | トレーサビリティが失われるため     |
| スコープ外の未記載                      | 境界不明確による手戻りを招くため   |
| 背景・期待効果の詳細を概要と二重管理    | 文書間不整合と保守負荷を招くため   |
