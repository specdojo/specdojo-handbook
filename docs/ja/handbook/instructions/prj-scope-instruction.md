# プロジェクトスコープ 作成指示

## 1. 目的と前提

- 目的: プロジェクトで実施する範囲と実施しない範囲を明確化し、境界条件を合意する。
- 参照ルール: `../rulebooks/prj-scope-rulebook.md`
- 主な内容: 対象業務、対象システム、対象期間、スコープ外、境界の判断基準、スコープ変更方針。

## 2. 入力情報

- プロジェクト名
- 対象業務（対象となる業務/プロセス）
- 対象システム（対象となるシステム/機能/利用者）
- 対象期間（初期リリース〜運用開始など）
- スコープ外（明示的にやらないこと、理由、任意で扱い時期）
- 境界の判断基準（任意）
- スコープ変更方針（任意）

## 3. 出力フォーマット

- Frontmatter は `id` / `type` / `status` を必須とする。
- `id` は `prj-<projectNo>-scope` を推奨し、`type` は `project` を使用する。
- 本文は次の見出し順で作成する。

1. 対象業務
2. 対象システム
3. 対象期間
4. スコープ外
5. 境界の判断基準（任意）
6. スコープ変更方針（任意）

## 4. 記述ルール

- 設計詳細（DB、API、画面詳細、テストケース列挙など）は記載しない。
- 対象業務は「現場で行う作業・判断の単位」で記述する。
- 対象システムは粒度を揃えて記述する。
- 対象期間が未確定の場合は `未確定` と明示し、決定条件を添える。
- スコープ外は曖昧にせず、理由を 1 行で添える。
- スコープ変更方針には、変更内容、トレードオフ、影響評価、承認者、記録方針を含める。

## 5. 禁止事項

- 曖昧語（`適切に`、`可能な限り`、`十分に` など）のみで結論を記載しない。
- スコープ外を未記載のまま確定しない。
- 粒度が揃わない機能列挙で判断を曖昧にしない。

## 6. 最終チェック

- 必須見出しが欠落していない。
- スコープ外に理由付きの項目がある。
- 境界判断または変更方針（任意章）に、運用可能な判断基準がある。
- 参照ルールの禁止事項に抵触していない。
- Markdown lint でエラーがない。
