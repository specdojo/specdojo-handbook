# SpecDojo ドキュメンテーション ポリシーガイド

SpecDojo Documentation Policy Guide

SpecDojo におけるドキュメンテーションの基本方針、設計原則、運用上の判断基準を定めるガイドです。

## 1. 基本方針

- **ID as Identity**:\
  ドキュメントは frontmatter の `id` をユニークな識別子として扱う。
  ファイル名やディレクトリは、可読性・分類・探索性を高めるための配置規約として扱う。

- **Convention over Configuration**:\
  個別設定を増やすのではなく、命名、配置、frontmatter、参照方法、章構成などの規約を優先する。

- **Don't Repeat Yourself**:\
  正本となる情報は一箇所に定義し、他のドキュメントでは参照する。
  ただし、読者の理解に必要な範囲で、要約・抜粋・リンク付き再掲は許容する。

- **Single Source of Truth**:\
  各情報には正本となるドキュメントを定める。
  派生ドキュメント、一覧、ビュー、レポートは正本から参照または生成する。

- **Structured Documentation**:\
  ドキュメントは、構造化された Markdown、YAML、JSON で記述する。
  Markdownは説明・判断・方針、YAMLは管理データ、JSONはツール連携や生成物に用いる。

- **Traceability**:\
  ドキュメント間の根拠、依存、派生、更新関係を frontmatter や構造化項目で明示する。

- **AI Readability**:\
  ドキュメントは、人間だけでなくAIが解析・生成・検証しやすいよう、
  見出し、frontmatter、表、ID、参照関係を一貫した形式で記述する。

- **Git as Source of Truth**:\
  ドキュメントの正本はGitリポジトリ上で管理する。
  Issue、チャット、会議メモ、外部ツールの内容は、必要に応じてGit上のドキュメントへ反映し、変更履歴を追跡可能にする。

## 2. 設計原則

SpecDojo のドキュメントは、単なる文章ではなく、プロジェクトを定義・管理・実行・検証するための構造化された情報資産として設計する。

### 2.1. ドキュメントは成果物として扱う

ドキュメントは、作業の補助資料ではなく、プロジェクトの成果物として扱う。

そのため、各ドキュメントには目的、責務、正本性、参照関係を明確に定める。

- 何を決定・説明・管理する文書かを明確にする。
- 他の文書と責務が重複しないようにする。
- 更新対象となる情報と、参照のみ行う情報を区別する。
- 成果物として管理すべき文書には、frontmatter で `id`、`type`、`status` などを付与する。

### 2.2. 正本と派生物を分離する

同じ情報を複数の場所で正として管理しない。

正本となるドキュメントと、そこから参照・要約・生成される派生ドキュメントを明確に分離する。

- 正本は、判断・承認・変更管理の対象とする。
- 派生物は、一覧、ビュー、レポート、生成物として扱う。
- 派生物に正本と異なる内容を直接記述しない。
- 自動生成された文書は、可能な限り `generated` として区別する。

### 2.3. 人間とAIの両方が読める構造にする

ドキュメントは、人間が理解しやすく、AIやツールが解析しやすい構造で記述する。

そのため、自然文だけに依存せず、見出し、frontmatter、表、リスト、ID、参照関係を一貫した形式で用いる。

- 章構成は、文書種別ごとの規約に従う。
- 重要な属性は frontmatter または構造化項目として記述する。
- 曖昧な表現だけでなく、判定可能な記述を心がける。
- AIが処理しやすいように、同じ意味の項目名や表現は統一する。

### 2.4. ファイル配置から文書の役割が推測できるようにする

ファイル名やディレクトリは、文書の正本識別子ではなく補助情報である。

ただし、人間がリポジトリを見たときに、文書の役割、分類、位置づけを推測できるように設計する。

- ディレクトリは、文書の責務やライフサイクルに基づいて分ける。
- ファイル名は、文書種別や対象を推測しやすい名前にする。
- 入口となる文書には `*-index.md` を用いる。
- 個別文書には、対象を表す語を含める。

### 2.5. 変更に強い参照構造にする

ドキュメント間の参照は、ファイル移動や再構成に耐えられるように設計する。

- 論理的な参照には `id` を用いる。
- 人間向けの本文リンクでは、必要に応じて相対パスを併用する。
- 根拠文書は `based_on` などの構造化項目で明示する。
- 置換・廃止・後継関係は `supersedes` などで追跡できるようにする。

### 2.6. 小さく分け、責務を明確にする

一つのドキュメントに多くの責務を持たせすぎない。

文書は、読みやすさ、更新しやすさ、レビューしやすさを保てる単位に分割する。

- 1文書には、原則として1つの主目的を持たせる。
- 大きくなりすぎる文書は、索引文書と個別文書に分割する。
- 分割した文書間では、正本の所在と参照関係を明確にする。
- ただし、過度な分割により読者の理解を妨げないようにする。

### 2.7. 生成可能な情報は生成物として扱う

一覧、集計、進捗ビュー、依存関係ビューなど、正本から機械的に導出できる情報は、可能な限り生成物として扱う。

- 手作業で二重管理しない。
- 生成元となる正本を明確にする。
- 生成物には、必要に応じて生成元、生成方法、生成日時を記録する。
- 生成物を直接編集する運用は避ける。

### 2.8. 最小限の必須項目から始める

ドキュメント体系は、最初から過度に複雑にしない。

必要な項目を定義しつつ、プロジェクトの規模や成熟度に応じて拡張できる設計にする。

- 必須項目は、識別、責務、状態、根拠、参照に関わるものを優先する。
- 任意項目は、必要になった段階で追加する。
- 小規模プロジェクトでも運用できる軽量な構成を基本とする。
- 大規模プロジェクトでは、同じ規約を保ったまま分割・拡張できるようにする。

## 3. 運用上の判断基準

SpecDojo のドキュメント運用では、判断に迷った場合、以下の基準に従う。

### 3.1. 正本性を優先する

同じ情報が複数の場所に存在する場合は、どのドキュメントを正本とするかを先に決める。

- 正本が決まっていない情報は、安易に複数文書へ記述しない。
- 既存の正本がある場合は、そこを更新する。
- 他の文書では、正本への参照、要約、抜粋にとどめる。
- 正本を変更する場合は、影響する派生物や参照先も確認する。

### 3.2. 規約で解決できるものは設定を増やさない

個別の例外設定を増やす前に、既存の命名規約、配置規約、章構成、frontmatter 規約で表現できないかを確認する。

- 一つの文書だけのために特別な項目を追加しない。
- 繰り返し発生する差異は、規約として整理する。
- 規約で扱えない明確な理由がある場合のみ、例外を認める。
- 例外を認める場合は、理由と影響範囲を記録する。

### 3.3. 重複よりも参照を優先する

同じ内容を再記述するよりも、正本への参照を優先する。

ただし、読者の理解に必要な要約や抜粋は許容する。

- 詳細は正本に記述する。
- 他の文書では、必要最小限の要約とリンクを記述する。
- 要約が正本と矛盾しないようにする。
- 要約が古くなりやすい場合は、要約ではなく参照に寄せる。

### 3.4. 読みやすさと構造化のバランスを取る

機械処理しやすさだけを優先して、人間が読みにくい文書にしない。

一方で、自由記述だけに依存して、AIやツールが解析できない文書にもさせない。

- 判断理由、背景、意図は Markdown の本文に記述する。
- 状態、分類、参照、依存関係は構造化項目として記述する。
- 表にした方が比較しやすい情報は表で記述する。
- 長い表や一覧は、必要に応じて YAML や生成物に分離する。

### 3.5. ファイル名よりもIDを信頼する

ドキュメントの同一性は、ファイル名ではなく frontmatter の `id` で判断する。

- ファイル名や配置が変更されても、同じ文書であれば `id` は維持する。
- 文書の意味や責務が大きく変わる場合は、新しい `id` を付与する。
- 旧文書との関係は `supersedes` などで明示する。
- 参照関係の検証では、原則として `id` を用いる。

### 3.6. 変更理由を残す

ドキュメントはGitで管理し、変更履歴を追跡可能にする。

重要な変更では、何を変えたかだけでなく、なぜ変えたかが分かるようにする。

- 変更理由は、コミットメッセージ、Pull Request、または関連文書に残す。
- 意思決定に関わる変更は、根拠文書や判断記録と関連付ける。
- 仕様、スコープ、成果物、スケジュールに影響する変更は、影響範囲を確認する。
- 一時的な議論やメモは、必要に応じて正式な文書へ反映する。

### 3.7. 迷ったら利用者の理解を優先する

ドキュメントの構成や分割に迷った場合は、最終的な利用者が理解しやすい形を優先する。

- 初めて読む人が文書の目的を理解できるかを確認する。
- 関係者が必要な情報にたどり着けるかを確認する。
- 参照が多すぎて読みにくい場合は、要約や索引を追加する。
- 詳細が多すぎて主旨が見えにくい場合は、個別文書へ分割する。

### 3.8. 手作業で同期し続ける運用を避ける

複数の文書を手作業で同期し続ける必要がある構成は避ける。

- 同じ情報を複数のファイルで更新しない。
- 一覧やレポートは、可能な限り正本から生成する。
- 生成できない場合でも、正本と派生情報の関係を明記する。
- 手作業で更新する必要がある場合は、更新タイミングと責任を明確にする。

### 3.9. 小さく始めて、必要に応じて拡張する

最初からすべての文書種別、項目、ビューを揃えることを目的にしない。

プロジェクトの規模、リスク、関係者数、変更頻度に応じて、必要な文書を追加する。

- 小規模プロジェクトでは、最小構成で開始する。
- 複雑性が増した場合に、文書を分割・追加する。
- 使われていない文書や項目は見直す。
- 追加した規約は、既存文書との整合性を確認する。

### 3.10. 例外は明示的に扱う

規約から外れる場合は、暗黙的に処理せず、例外として明示する。

- なぜ規約に従わないのかを記録する。
- 例外の適用範囲を限定する。
- 一時的な例外か、恒久的な例外かを区別する。
- 同じ例外が繰り返される場合は、規約自体の見直しを検討する。
