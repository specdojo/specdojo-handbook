# WBS から Schedule への展開戦略

WBS to Schedule Strategy

本ドキュメントは、`050-wbs/` 配下の WBS 定義を `060-schedule/` 配下のスケジュール定義へ落とし込む際のタスク分解方針、依存関係の考え方、タスク長の目安、`id` 命名規則を定義します。

## 1. 目的と適用範囲

- 対象は `docs/ja/sdh-ja-projects/prj-0001` の `wbs-*.yaml` と `sch-*.yaml`、および `sch-milestones.yaml` です。
- 対象とする入力は、`pm-wbs-decomposition-strategy.md` に従って WBS へ落とし込まれた Item のみです。
- WBS が定義する **WHAT（何を作るか）** を、Schedule では **WHEN（いつ）** と **ORDER（どの順序で行うか）** に展開します。
- WBS の分解・命名方針との整合は `pm-wbs-decomposition-strategy.md` を参照します。

## 2. 基本方針

| 観点       | 方針                                                                    |
| ---------- | ----------------------------------------------------------------------- |
| 展開単位   | `1 WBS Item = 1〜3 個程度の実行可能タスク` を基本とする                 |
| 対象範囲   | 成果物カタログで「WBSへの落とし込み対象外」とされた成果物は対象外とする |
| 実行性     | 各タスクは 1 回のエージェント実行または 1 回のレビューで完了可能にする  |
| 依存最小化 | 必須の先行条件のみ `depends_on` に記述し、不要な直列化を避ける          |
| 並列性     | 同時に進められるタスクは積極的に並列化し、`ready` なタスク数を確保する  |
| 判定可能性 | タスク完了は成果物レビューまたは検証結果で確認できる状態にする          |

## 3. WBS から Task への分解ルール

### 3.1. 基本ルール

- Schedule 化の対象は、WBS 定義ファイルに存在する Item のみとします。
- Schedule のタスクは **成果物そのもの** ではなく、**成果物を前進させる実行単位** とします。
- 1 つの WBS Item をそのまま 1 タスクにしてよいのは、1 回の実行で完了可能な場合のみです。
- `rules` → `instruction` → `sample` のように段階的な作業が必要な場合は、WBS 1 件を複数タスクへ分けます。

### 3.2. 推奨パターン

現在の `prj-0001` では、次の分解パターンを標準とします。

| パターン | 使いどころ                         | 例                             |
| -------- | ---------------------------------- | ------------------------------ |
| 1 タスク | 単独の軽量更新                     | メタデータ修正、軽微なレビュー |
| 2 タスク | 本体作成 + 検証/承認               | 単独ドキュメントの更新         |
| 3 タスク | `rules` → `instruction` → `sample` | ドキュメントファミリー整備     |

例:

- `WBS-PDT-BPS-0010` → `T-SDH-PDT-0010`, `T-SDH-PDT-0010-INS`, `T-SDH-PDT-0010-SMP`
- `WBS-PDT-CXD-0020` → ルール整備、Mermaid 整合、派生成果物整備

### 3.3. 分割する条件

次のいずれかに該当する場合は、タスクを分割します。

1. 1 回の実行で完了しない
2. 中間成果物にレビューゲートが必要
3. 担当ロールや専門性が途中で切り替わる
4. 依存関係を明確に切り出した方が、並列化や再計画がしやすい

### 3.4. WBS への落とし込み対象外の扱い

成果物カタログで「WBSへの落とし込み対象外」と明示された次の成果物は、本戦略の対象外です。

- 管理台帳: `pm-risk-register`, `pm-issue-log`, `pm-change-request-log`
- レポート: `pr-yyyy-mm-dd`, `mm-yyyy-mm-dd`
- 実行管理: `exec/`, `generated/`
- 決定記録: `dec-<NNNN>-<topic>`

これらに関する実行計画は、必要時に個別運用（定常運用計画、会議運営計画など）として管理し、本戦略の `sch-<scope>.yaml` には含めません。

## 4. タスク `id` 命名規則

### 4.1. 基本形式

タスク `id` は次の形式を基本とします。

```text
T-SDH-<DOMAIN>-<NNNN>
```

派生成果物フェーズを明示したい場合は、接尾辞を付与します。

```text
T-SDH-<DOMAIN>-<NNNN>-INS
T-SDH-<DOMAIN>-<NNNN>-SMP
```

例:

```text
T-SDH-PDT-0010
T-SDH-PDT-0010-INS
T-SDH-PDT-0010-SMP
```

### 4.2. 命名ルール

- `SDH` は `specdojo-handbook` プロジェクトの識別子として固定します。
- `<DOMAIN>` は `PRJ`, `AGT`, `PJD`, `PDT` のいずれかを使用し、WBS のドメイン略号と整合させます。
- `prj-deliverables-catalog.md` のドメインは管理分類、`<DOMAIN>` は実行計画（WBS/Schedule）分類として扱います。
- カタログドメインと WBS/Schedule 略号の対応は次表を基準とします。

| 成果物カタログのドメイン | 主な対象                                                                  | WBS/Schedule の `<DOMAIN>` |
| ------------------------ | ------------------------------------------------------------------------- | -------------------------- |
| `project`                | プロジェクト管理・計画（管理台帳・レポート・決定記録など対象外を除く）    | `PRJ`                      |
| `agent-customization`    | `.github` 配下の instructions / skills / prompts                          | `AGT`                      |
| `project-docs`           | `docs/ja/handbook/instructions` / `rules` / `samples`                     | `PJD`                      |
| `product-docs`           | 業務仕様・外部 IF・アーキ・システム設計・非機能・受入・テスト・移行・運用 | `PDT`                      |

- `<NNNN>` は 4 桁固定で `0010` 刻みを基本に採番します。
- 同一 WBS 内の派生タスクは近い番号帯にまとめます。
- 並び替えのみを理由に既存 `id` を変更しません。

## 5. 依存関係の考え方

### 5.1. 依存の原則

- `depends_on` には **本当に着手不能となる前提条件** のみを記述します。
- 「参考にしたい」「できれば先に見たい」程度の関係は依存にしません。
- 依存は原則として **Finish-to-Start** とし、複雑な関係を増やしすぎません。

### 5.2. 依存を置くべきケース

| 依存の種類         | 説明                             | 例                          |
| ------------------ | -------------------------------- | --------------------------- |
| 概念依存           | 先に上位概念や用語定義が必要     | `BPS` → `BR`、`GL` → 他仕様 |
| 成果物依存         | 入力となる成果物の完成が必要     | `BDD` → `BDS`               |
| 図表整合依存       | 図と本文の整合が必要             | `CDFD` → `CDSD` / `CSTD`    |
| 受入依存           | 受入基準は設計・仕様の後に整備   | `SF` → `BAC`                |
| マイルストーン依存 | プロジェクト開始・レビューゲート | `M-SDH-100` → 各初期タスク  |

### 5.3. 依存を最小化する判断基準

- その先行タスクが未完了でも、下流タスクを **80% 以上は進められる** なら依存を置かない
- 依存を追加すると待ち時間だけ増え、品質向上に寄与しない場合は依存を置かない
- 同一レビュー会でまとめて確認できるものは、細かい依存よりも同一フェーズ扱いを優先する

### 5.4. `prj-0001` における代表的な依存チェーン

| 領域         | 代表チェーン                           |
| ------------ | -------------------------------------- |
| 基盤         | `BPS` → `BR` → `BES`                   |
| データ       | `BDD` → `BDS`                          |
| リファレンス | `GL` と `REFDATA` は並列開始可能       |
| モデル       | `CDFD` → `CDSD` / `CSTD` → `CCD`       |
| システム     | `BPS` / `UIS` を参照しつつ `SF` を整備 |
| 受入         | `SF` → `BAC`                           |

## 6. タスク長の考え方

### 6.1. 基本目安

`duration_days` は次を目安とします。

| 長さ    | 目安      | 用途                                      |
| ------- | --------- | ----------------------------------------- |
| `0.125` | 約 1 時間 | 軽微な修正、確認、単独レビュー            |
| `0.25`  | 半日未満  | `instruction` / `sample` 単体の作成・更新 |
| `0.5`   | 半日程度  | `rules` や主要本文の起草・レビュー        |
| `1.0`   | 1 日程度  | 複数成果物を含むファミリー整備            |

### 6.2. 長すぎるタスクの扱い

次のいずれかに該当する場合は分割を検討します。

1. `duration_days` が `1.0` を超える
2. 変更対象ファイルが 5 件を大きく超える
3. 途中でレビュー待ちや意思決定待ちが発生する
4. 担当者が切り替わる

### 6.3. 短すぎるタスクの扱い

- `0` 日タスクは原則禁止とします。
- 10〜15 分程度の作業は、前後の実行タスクへ吸収して 1 つにまとめます。
- タスクを細かくしすぎて依存線だけが増える状態は避けます。

## 7. マイルストーンとフェーズの考え方

- `sch-milestones.yaml` には、開始ゲート・レビューゲート・完了ゲートを定義します。
- ドメインスケジュールの初期タスクは、必要に応じて開始マイルストーンへ依存させます。
- Phase は、`rules` 本体整備と派生成果物整備のような **意味のある区切り** に限定します。
- Phase は説明用の概念であり、WBS と重複して粒度を増やしすぎません。

## 8. タスク項目の記述ルール

| 項目            | 記述ルール                             |
| --------------- | -------------------------------------- |
| `id`            | 一意で、人が追跡しやすい形式にする     |
| `wbs`           | 必ず対応する WBS Item を参照する       |
| `name`          | 作業内容が一読で分かる短いラベルにする |
| `duration_days` | 実行可能な長さで見積もる               |
| `depends_on`    | 真に必要な先行タスクだけを列挙する     |
| `owner`         | 主責任者ロールを設定する               |
| `tags`          | フィルタ・集計しやすい分類を付ける     |
| `notes`         | フェーズや運用前提など補足事項を書く   |

## 9. 具体例

```yaml
tasks:
  - id: T-SDH-PDT-0010
    wbs: WBS-PDT-BPS-0010
    name: 業務プロセス仕様ルールの起草・レビュー
    duration_days: 0.5
    depends_on:
      - M-SDH-100
    owner: BA

  - id: T-SDH-PDT-0010-INS
    wbs: WBS-PDT-BPS-0010
    name: 業務プロセス仕様 instruction.md の起草・レビュー
    duration_days: 0.25
    depends_on:
      - T-SDH-PDT-0010
    owner: BA

  - id: T-SDH-PDT-0010-SMP
    wbs: WBS-PDT-BPS-0010
    name: 業務プロセス仕様 sample.md の起草・レビュー
    duration_days: 0.25
    depends_on:
      - T-SDH-PDT-0010-INS
    owner: BA
```

この例では、1 つの WBS Item を **本体整備 → instruction → sample** の 3 タスクに分け、依存を最小限に保ちながら実行順序を明確にしています。

## 10. アンチパターン

- `depends_on` を増やしすぎて全タスクが直列になる
- `duration_days: 0` や `duration_days: 3` のように極端な値を使う
- WBS と Schedule の `id` 体系が対応しておらず、追跡できない
- 仕様上は独立なのに、レビュー都合だけで機械的に全部を直列化する
- `name` が「対応」「更新」だけで、何をするタスクか判別できない

## 11. 運用メモ

- WBS を追加・変更した場合は、必ず対応する `sch-<scope>.yaml` のタスク化方針を見直します。
- 成果物カタログの「WBSへの落とし込み対象外」定義が変更された場合は、先に WBS 側の対象範囲を更新し、その後本戦略へ反映します。
- 依存関係に迷う場合は、まず **依存を置かない案** を考え、着手不能な理由が明確な場合のみ追加します。
- タスク長に迷う場合は、**1 回のエージェント作業で完了確認まで到達できるか** を基準にします。
- 本戦略の更新後は、Schedule の妥当性を `ready` タスク数と依存の深さで点検します。
