# WBS分解戦略

WBS Decomposition Strategy

本ドキュメントは、`030-prj-deliverables-catalog.md` に定義した成果物を `050-wbs/` 配下の WBS 定義へ落とし込む際の分解粒度、ファイル分割方針、`id` 命名規則、記述ルールを定義します。

## 1. 目的と適用範囲

- 対象は `prj-0001` の成果物カタログと WBS 定義ファイルです。
- 本戦略は `050-wbs/` 配下の各 `wbs-<scope>.yaml` と、それに対応する `060-schedule/` 配下の `sch-<scope>.yaml` に適用します。
- WBS は **WHAT（何を作るか）** を定義し、**WHEN（いつ行うか）** と **ORDER（どの順序で行うか）** は Schedule で管理します。
- 共通原則は `docs/ja/handbook/guidelines/specdojo-wbs-design-guide.md` を上位ルールとして参照します。

## 2. 基本方針

| 観点             | 方針                                                               |
| ---------------- | ------------------------------------------------------------------ |
| 分解基準         | `1 WBS Item = 1 完了判定を共有する成果物ファミリー` を基本とする   |
| トレーサビリティ | 各 WBS Item は成果物カタログの対象行または対象グループに対応づける |
| 安定性           | `id` は人が読める略号を使い、途中で意味なく変更しない              |
| 判定可能性       | `done_criteria` はレビューで完了可否を判定できる文章にする         |
| 役割分担         | WBS はスコープ、Schedule は順序・依存・日程を定義する              |

## 3. WBS ファイルへの振り分け方針

成果物カタログのドメイン・性質に応じて、WBS は次の単位でファイル分割します。

| 成果物カタログ上の領域                       | WBS ファイル                   | `id` のドメイン略号 | 分割方針                                                   |
| -------------------------------------------- | ------------------------------ | ------------------- | ---------------------------------------------------------- |
| プロジェクト管理、移行、運用設計、メタルール | `wbs-governance.yaml`          | `GOV`               | ガバナンス・運営方針としてまとめる                         |
| エージェントカスタマイズ                     | `wbs-agent-customization.yaml` | `AGT`               | instructions / skills / prompts などの運用セットでまとめる |
| 業務仕様                                     | `wbs-business.yaml`            | `BIZ`               | 業務仕様の文書ファミリー単位で分解する                     |
| 外部 IF 仕様、アーキテクチャ、システム設計   | `wbs-design.yaml`              | `DES`               | 設計成果物ファミリー単位で分解する                         |
| 業務受入条件、システム受入条件、テスト       | `wbs-quality.yaml`             | `QLTY`              | 品質保証・受入判定の観点でまとめる                         |
| 非機能要件                                   | `wbs-nfr.yaml`                 | `NFR`               | 非機能カテゴリ単位で分解する                               |

## 4. WBS 粒度ルール

### 4.1. 基本粒度

- 基本単位は **単一ファイル** ではなく、**同じ完了条件でレビュー・承認できる成果物ファミリー** とします。
- `rules` / `instruction` / `sample` の 3 点が常に一体で更新される場合は、**1 WBS Item にまとめる** ことを推奨します。
- Markdown 本文と Mermaid 図のように、整合した 1 セットとして管理すべきものも **1 WBS Item** とします。

### 4.2. 分割する条件

次のいずれかに該当する場合は、WBS Item を分割します。

1. 完了条件が別であり、独立してレビューできる
2. 担当ロール（`owner`）や専門性（`component`）が異なる
3. スケジュール上の依存や優先度が大きく異なる
4. 成果物の数が多く、1 Item ではスコープが曖昧になる

### 4.3. まとめてよい条件

次のいずれかに該当する場合は、WBS Item をまとめてよいものとします。

1. 同一テーマの成果物群で、完了判定を 1 つにできる
2. 常に同じライフサイクルで更新される
3. 1 つの成果物群として利用者が理解する方が自然である

例:

| 対象                                                    | 推奨                                         |
| ------------------------------------------------------- | -------------------------------------------- |
| `bps-rules.md` / `bps-instruction.md` / `bps-sample.md` | 1 Item にまとめる                            |
| `cxd` と `cxd-mermaid`                                  | 1 Item にまとめる                            |
| `cld` / `stsd` / `sld`                                  | リファレンスデータ群として 1 Item にまとめる |
| 独立した受入条件とテスト計画                            | 別 Item に分割する                           |

## 5. `id` 命名規則

### 5.1. 基本形式

`id` は次の形式を基本とします。

```text
WBS-<DOMAIN>-<ARTIFACT>-<NNN>
```

例:

```text
WBS-BIZ-BPS-010
WBS-DES-IFX-010
WBS-QLTY-UTC-030
WBS-GOV-PRJ-010
```

### 5.2. ドメイン略号

| 略号   | 意味                        |
| ------ | --------------------------- |
| `GOV`  | Governance                  |
| `AGT`  | Agent Customization         |
| `BIZ`  | Business                    |
| `DES`  | Design                      |
| `QLTY` | Quality                     |
| `NFR`  | Non-Functional Requirements |

### 5.3. ARTIFACT の決め方

- 原則として成果物カタログの `name` に対応する短い識別子を使います。
- 既存の略号がある場合はそれを優先します。例: `BPS` `BR` `BDD` `IFX` `UTC`
- 複数成果物を 1 Item にまとめる場合は、代表的なファミリー名を使います。例: `REFDATA`
- 略号は **意味が推測できること** を優先し、過度に短縮しません。

### 5.4. 連番の付け方

- 連番は `010` 刻みで付与します。
- 同一ファイル内で上から順に `010`, `020`, `030` と採番します。
- 途中追加が想定されるため、`001` 刻みではなく `010` 刻みを維持します。
- 並び順変更のみを理由に既存 `id` を付け替えません。

## 6. WBS Item 記述ルール

| 項目            | 記述ルール                                           |
| --------------- | ---------------------------------------------------- |
| `name`          | 人が一覧で識別できる短い日本語ラベルにする           |
| `description`   | 何を対象にし、どの成果物群を整備するのかを具体化する |
| `owner`         | 主責任ロールを 1 つ設定する                          |
| `component`     | 専門領域を短い英語で表す                             |
| `deliverables`  | 実際に作成・更新するファイルパスを列挙する           |
| `done_criteria` | 完了状態をレビュー可能な文章で記述する               |

`done_criteria` の良い例:

```text
BPS ルール・指示ペアが、プロセス仕様の必須セクション・禁止事項・AI ガイダンスを一貫して定義していること。
```

`done_criteria` の悪い例:

```text
BPS を作成する
```

## 7. 成果物カタログから WBS への落とし込み手順

1. `030-prj-deliverables-catalog.md` から対象の成果物または成果物群を特定する
2. 既存の `wbs-<scope>.yaml` に同じファミリーがあるか確認する
3. 新規 Item とするか、既存 Item に統合するかを判断する
4. `WBS-<DOMAIN>-<ARTIFACT>-<NNN>` 形式で `id` を付与する
5. `deliverables` に実ファイルパスを列挙する
6. `done_criteria` を成果物レビュー観点で記述する
7. 必要に応じて対応する `sch-<scope>.yaml` にタスク・依存を追加する

## 8. 具体例

| 成果物カタログ上の対象                               | WBS Item の考え方                        | 例                    |
| ---------------------------------------------------- | ---------------------------------------- | --------------------- |
| `bps`                                                | 単独ファミリーとして 1 Item              | `WBS-BIZ-BPS-010`     |
| `cld` `stsd` `sld`                                   | リファレンスデータ群として 1 Item        | `WBS-BIZ-REFDATA-070` |
| `cxd` と `cxd-mermaid`                               | 文書と図の整合セットとして 1 Item        | `WBS-DES-CXD-020`     |
| `prj-overview` `prj-scope` `prj-issues-and-approach` | プロジェクト文書ファミリーとしてまとめる | `WBS-GOV-PRJ-010`     |

## 9. アンチパターン

- 1 Item に「業務仕様全部」のような広すぎるスコープを入れる
- `rules` と `instruction` を常に同時更新する前提なのに不必要に分割する
- `done_criteria` を「更新する」「整備する」だけで終わらせる
- `id` に意味の分からない略号を使う
- Schedule で管理すべき日付や順序を WBS の `description` に書き込む

## 10. 運用メモ

- 新しい成果物種別を成果物カタログへ追加した場合は、まず本戦略に照らして **どの WBS ファイルへ所属させるか** を決めます。
- 分解判断に迷う場合は、**完了条件を 1 つに書けるか** を基準にします。
- 本戦略を更新した場合は、関連する `wbs-*.yaml` と `sch-*.yaml` の整合を確認します。
