# プロジェクト憲章: 駄菓子屋きぬや 販売管理システム

## 1. 本書の目的

本書は、駄菓子屋きぬや 販売管理システム構築プロジェクトの立ち上げを正式に認可し、プロジェクトマネージャーに詳細計画の策定と初期準備を進める権限を付与する文書である。

本書の承認は「立ち上げ認可」であり、本格実行開始の認可ではない。本格実行開始は、詳細計画（`pm-plan`）の完成と GO / Not GO 判断を経て別途決定する。

## 2. 認可対象

| 項目                     | 内容                                                                                 |
| ------------------------ | ------------------------------------------------------------------------------------ |
| プロジェクト名           | 駄菓子屋きぬや 販売管理システム構築                                                  |
| プロジェクト ID          | prj-0001                                                                             |
| 認可対象                 | 立ち上げフェーズ（要件整理・構成方針策定・詳細計画策定）の開始                       |
| プロジェクト概要         | 販売・在庫・常連顧客（つけ）の管理を一元化し、店主一人でも運用できる仕組みを構築する |
| スポンサー               | 店主（きぬや経営者）                                                                 |
| プロジェクトマネージャー | 開発支援リーダー                                                                     |

## 3. プロジェクトの目的

- 販売・在庫・つけ管理をシンプルな仕組みで一元化し、ノートと記憶に依存した運用から脱却する。
- 店主一人でも無理なく操作できる画面を提供し、欠品・在庫ムラ・つけ漏れを削減する。
- 詳細な成功指標と測定方法は `prj-success-criteria-and-acceptance-criteria` で定義する。

## 4. ハイレベルスコープ

| 区分   | 内容                                                      | 詳細化先               |
| ------ | --------------------------------------------------------- | ---------------------- |
| 対象   | 単一店舗の販売記録、在庫管理、常連顧客（つけ）管理        | `prj-scope`            |
| 対象外 | EC 連携、複数店舗管理、本格会計連携、仕入れ先システム連携 | `prj-scope`            |
| 未確定 | オフライン時の運用方針、仕入れ先への発注通知              | `prj-scope` / 未決事項 |

## 5. ハイレベル成果物

| 成果物       | 概要                                 | 詳細化先                   |
| ------------ | ------------------------------------ | -------------------------- |
| 販売記録機能 | 日々の販売内容と売上を記録する       | `prj-deliverables-catalog` |
| 在庫管理機能 | 商品在庫の確認・更新を行う           | `prj-deliverables-catalog` |
| つけ管理機能 | 顧客ごとの残高確認・更新を行う       | `prj-deliverables-catalog` |
| 管理画面     | 店主がタブレット 1 台で操作できる UI | `prj-deliverables-catalog` |

## 6. ハイレベル成功基準

| 基準             | 概要                              | 詳細化先                                       |
| ---------------- | --------------------------------- | ---------------------------------------------- |
| 欠品削減         | 人気商品の欠品回数を月間 50% 削減 | `prj-success-criteria-and-acceptance-criteria` |
| 仕入れムダ削減   | 廃棄・滞留コストを年間 20% 削減   | `prj-success-criteria-and-acceptance-criteria` |
| つけ確認の迅速化 | 顧客検索から残高表示まで 1 分以内 | `prj-success-criteria-and-acceptance-criteria` |

## 9. 主要前提・制約

| 区分 | 内容                                                                           |
| ---- | ------------------------------------------------------------------------------ |
| 前提 | 単一店舗での運用を前提とする                                                   |
| 前提 | 操作端末は店内のタブレット 1 台（_ASSUMPTION_: iOS または Android タブレット） |
| 前提 | 現行の販売手順・接客フローを大きく変えない                                     |
| 制約 | 費用上限は _UNDECIDED_（詳細計画策定後に確定）                                 |
| 制約 | リリース目標日は _UNDECIDED_（詳細計画策定後に確定）                           |
| 制約 | オフライン時の挙動は対象外（初期リリース後に検討）                             |

## 11. 本格実行開始の GO / Not GO 判断

本格実行開始は、`pm-plan` の完成後に次の観点で判断する。

| 判断観点         | 確認内容                               |
| ---------------- | -------------------------------------- |
| 費用合意         | 費用上限と概算見積もりが合意されている |
| スケジュール合意 | リリース目標日が店主と合意されている   |
| リスク許容       | 主要リスクと対応方針が合意されている   |
| 体制確認         | 実行主体（メンバー）が確定している     |

判断結果は GO / Not GO 決定記録として保存し、`pm-plan` または別途管理する決定記録に記録する。
