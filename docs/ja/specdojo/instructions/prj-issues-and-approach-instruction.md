# プロジェクト課題と解決アプローチ 作成指示

## 1. 目的と前提

- 目的: 課題、原因、解決策候補、採用アプローチを整理し、判断理由を明文化する。
- 参照ルール: `../rulebooks/prj-issues-and-approach-rulebook.md`
- 主な内容: 課題一覧、原因、解決策候補、採用アプローチと理由、トレードオフ/リスク、次の検討事項。

## 2. 入力情報

- プロジェクト名
- 課題一覧（課題、影響、優先度）
- 原因（事実と仮説を区別）
- 解決策候補（最低 2 案）
- 採用アプローチ（採用案、採用理由、採用しない案と理由）
- トレードオフ/リスク（任意）
- 次の検討事項（ToDo）（任意）

## 3. 出力フォーマット

- Frontmatter は `id` / `type` / `status` を必須とする。
- `id` は `prj-<projectNo>-issues-and-approach` を推奨し、`type` は `project` を使用する。
- 本文は次の見出し順で作成する。

1. 課題一覧
2. 原因（仮説でも可）
3. 解決策候補
4. 採用アプローチと理由
5. トレードオフ/リスク（任意）
6. 次の検討事項（ToDo）（任意）

## 4. 記述ルール

- 設計詳細（DB、API、画面詳細、テストケース列挙など）は記載しない。
- 課題は「困っている現象」として記載し、解決策の先取り表現は避ける。
- 課題には最低限「影響」と「優先度」を含める。
- 原因は `事実` と `仮説` を区別して記載する。
- 解決策候補は最低 2 案を提示し、採用案の判断軸（効果/コスト/期間/リスク/現場適合）を示す。
- 採用しない案がある場合は理由を記載する。
- トレードオフ/リスクは「起きうること」と「軽減策」をセットで記載する。

## 5. 禁止事項

- 曖昧語（`適切に`、`可能な限り`、`十分に` など）のみで結論を記載しない。
- 候補を 1 案のみで確定し、比較可能性を失わせない。
- 判断軸や理由を欠いた採用判断を記載しない。

## 6. 最終チェック

- 必須見出しが欠落していない。
- 課題一覧に「影響」と「優先度」が含まれている。
- 解決策候補が 2 案以上あり、採用理由が明示されている。
- 参照ルールの禁止事項に抵触していない。
- Markdown lint でエラーがない。
