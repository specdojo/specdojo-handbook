# 成果物カタログ（ドメイン別）作成指示

## 1. 目的と前提

- **目的**: `dct-<domain>.md`（ドメイン別成果物カタログ）を 1 ファイル生成する。
- **対象**: 特定ドメインに属する成果物の台帳。ID・種別・根拠・配置先・概要のメタ情報を一覧化する。
- **対象外**: 成果物の詳細本文（仕様・手順・構成図など）。詳細は成果物本体ファイルに集約する。
- `dct-index` の「共通ルール」を SSOT として参照し、重複定義を避ける。

## 2. 入力情報

以下の情報を事前に確認してください。不明な場合は `_UNDECIDED_:` として本文に明記してよい。

- `project-id`（例: `prj-0001`）
- ドメイン識別子（英小文字・kebab-case。例: `project-definition`）
- 親となる `dct-index` の `id`
- 対象ドメインの成果物一覧（`local-id`・種別・根拠・配置先・概要）
- サブドメインに分かれる場合はサブドメインの区分と各配置先

## 3. 出力フォーマット

### Frontmatter（必須）

```yaml
---
id: <project-id>:dct-<domain>
type: project
status: draft # draft / ready / deprecated のいずれか
part_of:
  - <project-id>:dct-index
rulebook: dct-rulebook
---
```

- `id` は `<project-id>:dct-<domain>` 形式（例: `prj-0001:dct-project-definition`）。
- `part_of` に親 `dct-index` の `id` を配列形式で記載する（必須）。
- `based_on` は直接根拠がある場合のみ記載する。

### 本文構成

本文冒頭に以下のメタ情報を箇条書きで記載する（`配置先` は必須）。

```markdown
- project-id: prj-0001
- ドメイン: project-definition
- 配置先: docs/ja/projects/prj-0001/020-project-definition/
```

**章構成パターン:**

| パターン | 適用条件                      | 構成                                                          |
| -------- | ----------------------------- | ------------------------------------------------------------- |
| 平坦型   | サブドメインに分かれない場合  | 先頭メタ情報 + 成果物一覧テーブル                             |
| 章分割型 | サブドメインが2つ以上ある場合 | 先頭メタ情報 + サブドメイン別章（各章に `配置先` + テーブル） |

## 4. 記述ルール

### タイトルと概要

- H1 はドキュメントの役割が一読で判別できる名称とし、「成果物カタログ `<ドメイン>`」のように対象と目的を含めて記述する。
- H1 の直下に英語名を 1 行で記載する（例: `Project Deliverables Catalog: Project Definition`）。
- 英語名の直下に、ドキュメント概要を 2〜3 行で記述する。概要には「対象」「目的」「扱う範囲」を含める。詳細な手順や実装説明は書かない。

### 成果物一覧テーブルの必須列

| 列名       | 説明                                                              | 必須 |
| ---------- | ----------------------------------------------------------------- | ---- |
| `local-id` | 成果物の論理名（英小文字・kebab-case）。backtick で囲んで記載する | ○    |
| `成果物名` | 業務ユーザーが理解可能な日本語名                                  | ○    |
| `種別`     | `work` / `control` / `generated` のいずれか                       | ○    |
| `根拠`     | 直接依存する `local-id`（カンマ区切り。依存なしは `-`）           | ○    |
| `概要`     | 成果物の目的を1文で記述                                           | ○    |

任意列:

- `ARTIFACT`: プロジェクト内で成果物コードを管理する慣習がある場合のみ追加する。不要な場合は列ごと省略する。

### 先頭メタ情報

- `project-id`・`ドメイン`・`配置先` は必須。箇条書きで記載する。
- `配置先` はドメイン単位の既定ディレクトリをリポジトリルートからのパスで記載する（ファイル単位パスは WBS で管理）。
- `DOMAIN`（略称）はプロジェクト内で短縮コードを使う慣習がある場合のみ記載する。

### 章分割型の記述

- サブドメイン別に章（`##` または `###`）を設け、章タイトルはサブドメインの業務名称とする。
- 各章の冒頭に `配置先` を箇条書きで記載する。
- WBS 展開対象外の `control` / `generated` 成果物が含まれる場合は引用ブロック（`>`）で注記する。

### 種別の扱い

- `work`: WBS 展開対象。成果物を実際に作成する作業に対応する。
- `control`: プロジェクト管理・統制文書。WBS 展開対象外。
- `generated`: 自動生成・派生成果物。WBS 展開対象外。

## 5. 禁止事項

- `local-id` に日本語・空白・大文字を使用しない。
- 成果物の詳細本文（仕様・手順・構成図など）を `dct-<domain>` に記載しない。
- 種別を `work` / `control` / `generated` 以外に拡張しない。
- 同一 `local-id` を複数ドメインのカタログに重複して記載しない。
- `根拠` 列に `local-id` 以外の表記（ファイルパス・URL など）を混在させない。

- [ ] H1 直下に英語名が 1 行で記載されている
- [ ] 英語名直下に概要が 2〜3 行で記載され、「対象・目的・扱う範囲」が含まれている
- [ ] Frontmatter の `id` が `<project-id>:dct-<domain>` 形式になっている
- [ ] `part_of` に親 `dct-index` の `id` が記載されている
- [ ] 先頭メタ情報に `project-id`・`ドメイン`・`配置先` が含まれている
- [ ] 成果物一覧テーブルに必須5列（`local-id`・`成果物名`・`種別`・`根拠`・`概要`）が含まれている
- [ ] `local-id` がすべて英小文字・kebab-case になっている
- [ ] `種別` がすべて `work` / `control` / `generated` のいずれかになっている
- [ ] `根拠` が `local-id` のみで記載されている（ファイルパス混在なし）
