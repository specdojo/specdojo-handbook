# プロジェクト憲章 作成指示

## 1. 目的と前提

- 目的: プロジェクトの立ち上げを正式に認可し、プロジェクトマネージャーに詳細計画の策定と初期準備を進める権限を付与する。
- 参照ルール: `../rulebooks/prj-charter-rulebook.md`
- 主な内容: 認可対象、ハイレベルスコープ/成果物/成功基準、権限委譲、GO/Not GO 判断、承認と証跡。
- 本憲章の承認は「立ち上げ認可」であり、本格実行開始の認可ではない。
- `prj-overview` と `prj-stakeholder-register` を入力として使い、後続の定義・管理計画文書群の起点となる。

## 2. 入力情報

- `prj-overview`（背景・目的・必要性・期待効果・前提条件）
- `prj-stakeholder-register`（初期ステークホルダー、主要役割）
- スポンサーおよびプロジェクトマネージャーの特定
- 権限委譲の範囲（PM に付与する権限とスポンサー承認が必要な事項）
- 立ち上げ時点で確定済みの前提・制約
- 承認者・承認日・証跡リンク（議事録、Issue、PR など）
- 未決事項の一覧（今後の意思決定が必要な論点）

## 3. 出力フォーマット

- Frontmatter に `id` / `type` / `status` / `rulebook` を含める。

```yaml
---
id: <project-id>-prj-charter
type: project
status: draft
rulebook: prj-charter-rulebook
based_on:
  - <project-id>-prj-overview
  - <project-id>-prj-stakeholder-register
---
```

- 本文は次の順序で作成する（順序固定）。

1. 本書の目的
2. 認可対象
3. プロジェクトの目的
4. ハイレベルスコープ
5. ハイレベル成果物
6. ハイレベル成功基準
7. 初期ステークホルダー
8. 権限委譲
9. 主要前提・制約
10. 後続で作成・詳細化する文書
11. 本格実行開始の GO / Not GO 判断
12. 承認
13. 未決事項（任意）

- 各章に次の推奨表を配置する。

**認可対象:**

| 項目                 | 内容 |
| -------------------- | ---- |
| プロジェクト名       |      |
| プロジェクトID       |      |
| 認可対象             |      |
| プロジェクト概要     |      |
| 初期ステークホルダー |      |

**ハイレベルスコープ:**

| 区分   | 内容 | 詳細化先               |
| ------ | ---- | ---------------------- |
| 対象   |      | `prj-scope`            |
| 対象外 |      | `prj-scope`            |
| 未確定 |      | `prj-scope` / 未決事項 |

**権限マトリクス:**

| 項目         | 決裁者 | 実行責任者 | 協議先 | 証跡 |
| ------------ | ------ | ---------- | ------ | ---- |
| 立ち上げ認可 |        |            |        |      |
| 詳細計画策定 |        |            |        |      |
| 本格実行開始 |        |            |        |      |
| 主要変更     |        |            |        |      |

**承認履歴:**

| 版  | 承認日 | 承認者 | 承認対象 | 証跡リンク |
| --- | ------ | ------ | -------- | ---------- |

## 4. 記述ルール

- 「本書の目的」で本憲章が立ち上げ認可文書であり、本格実行開始の承認でないことを明記する。
- 背景・目的・期待効果の詳細は `prj-overview` に委ね、憲章には認可判断に必要な要約のみを記載する。
- ハイレベルスコープ・成果物・成功基準は概要レベルに留め、詳細化先の文書名（`prj-scope`、`prj-deliverables-catalog`、`prj-success-criteria-and-acceptance-criteria`）を必ず明示する。
- 未確定事項は `_UNDECIDED_`、仮置き前提は `_ASSUMPTION_` で明示する。
- 承認履歴には承認者・承認日・承認対象・証跡リンクを必ず含める。口頭合意のみで完了扱いにしない。
- GO / Not GO 判断は結果を `pm-plan` または決定記録に記録する旨を明示する。
- `pm-plan`、`pm-organization` は憲章承認後に作成する文書であり、`based_on` に含めない。

## 5. 禁止事項

- 詳細スコープ・成果物一覧・具体的な成功指標を憲章に過剰記載する（`prj-scope` 等との二重管理を招く）。
- 憲章承認を本格実行開始の承認と混同する。
- 承認者未記載・証跡リンクなしで正式認可と扱う。
- 設計詳細（DB定義、API項目詳細等）を記載する。
- 背景・期待効果の詳細を `prj-overview` と二重管理する。
- GO / Not GO 判断の記録先を定義しない。
- 後続文書の未作成を隠して確定済みのように記載する。
- 曖昧語（`適切に`、`十分に`）のみで成功基準を記述する。

## 6. 最終チェック

- 12 章（任意の「未決事項」を除く）がすべて存在し、順序が固定されている。
- 「本書の目的」に本格実行開始の承認ではない旨が明記されている。
- ハイレベルスコープ・成果物・成功基準のそれぞれに詳細化先の文書名がある。
- 権限マトリクスに承認者・実行責任者が設定されている。
- 承認履歴に承認者・承認日・証跡リンクが含まれている。
- `based_on` に `pm-plan`、`pm-organization` が含まれていない。
- 参照ルールの禁止事項に抵触していない。
- Markdown lint でエラーがない。
