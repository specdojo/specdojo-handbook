# 品質管理計画 作成ルール

Quality Management Plan Documentation Rules

本ドキュメントは、`品質管理計画` を一貫した粒度で作成するためのルールを定義する。

## 1. 全体方針

- プロジェクトにおける品質目標、レビュー方針、品質メトリクス、検査基準、是正プロセスを定義する。
- 品質確認の実作業は `QE` が担い、最終判断は `PO` が行うことを明記する。
- 品質管理上の責任主体は Role code で記述し、`pm-organization` で採用済みの Role code のみを使う。
- AI Agent は作業支援・レビュー支援を担ってよいが、最終判断や公開可否判断は人間が行うことを明記する。
- 品質メトリクスは算出方法・閾値・計測頻度・報告先を定量的に記述する。曖昧な目標値のみの記載は不可。

## 2. 位置づけと用語定義

| ドキュメント                 | 役割                                                                                 |
| ---------------------------- | ------------------------------------------------------------------------------------ |
| `pm-plan`                    | 全体管理方針。品質・リスク・課題管理の上位方針を定義する                             |
| `pm-organization`            | 採用ロールと未採用ロールを定義する。品質管理上の責任主体の正とする                   |
| `pm-raci`                    | 成果物・プロセスごとの責任分担を定義する                                             |
| `pm-quality-management-plan` | 品質目標・レビュー方針・メトリクス・是正プロセスの詳細を定義する                     |
| `generated` 成果物           | 正本（work / control）から派生する補助成果物。正本と矛盾した場合は正本を先に修正する |

## 3. ファイル命名・ID規則

- ファイル名は `pm-quality-management-plan.md` を推奨する。
- 推奨パス: `docs/ja/projects/<project-id>/030-project-management/010-management-plan/pm-quality-management-plan.md`
- `id` は `<project-id>:pm-quality-management-plan` 形式を推奨する。例: `prj-0001:pm-quality-management-plan`

## 6. 記述ガイド

### 6.1. 概要

- 品質管理の目的、適用範囲、前提となる体制（`pm-organization` 採用ロール）を 2〜4 行で示す。
- AI Agent の役割（支援のみ、最終判断は人間の `PO`）を明記する。

### 6.2. 品質目標

- 項目・目標値・期限/タイミング・判定方法・責任ロールを表で示す。
- 目標値は定量または判定可能な条件で記述し、定性のみにしない。

推奨フォーマット:

| 項目 | 目標値 | 期限 / タイミング | 判定方法 | 責任ロール |
| ---- | ------ | ----------------- | -------- | ---------- |

### 6.3. レビュープロセス

- 「3.1 レビュー種別」では種別・主な対象・実施ロール・頻度/タイミング・出口条件を表で示す。

推奨フォーマット:

| 種別 | 主な対象 | 実施ロール | 頻度 / タイミング | 出口条件 |
| ---- | -------- | ---------- | ----------------- | -------- |

- 「3.2 レビュー手順」では番号付き手順で記述する。重大指摘の最終確認者（`PO`）を明示する。

### 6.4. 品質メトリクス

- 指標・算出方法・閾値・計測頻度・報告先を明記する。ツールコマンドが確定している場合は記載してよい。

推奨フォーマット:

| 指標 | 算出方法 | 閾値 | 計測頻度 | 報告先 |
| ---- | -------- | ---- | -------- | ------ |

### 6.5. 検査基準と是正

- 「5.1 検査基準」では観点・受入基準を表で示す（構造・内容・整合・追跡・公開・AI 利用など）。
- 「5.2 是正プロセス」では不適合区分・例・是正方法・再判定者を表で示す。

推奨フォーマット（是正プロセス）:

| 不適合区分 | 例  | 是正方法 | 再判定 |
| ---------- | --- | -------- | ------ |

- 「5.3 生成物の扱い」では `generated` 成果物と正本の関係、矛盾時の対処方針を記述する。

### 6.6. 品質上の役割分担

- `pm-organization` で採用済みの Role code を行として、品質管理上の責務を記述する。
- 未採用 Role code の行は含めない。

推奨フォーマット:

| Role code | 品質管理上の責務 |
| --------- | ---------------- |

### 6.7. 関連ドキュメント

- `pm-plan`、`pm-organization`、`pm-raci`、`pm-communication-plan`、成果物カタログ、プロジェクト登録簿、リスク・課題・変更要求ログへの導線を記載する。

### 6.8. 見直し条件

- 本書を更新するトリガー・見直し内容・責任ロールを表で示す。

推奨フォーマット:

| 更新トリガー | 見直し内容 | 責任ロール |
| ------------ | ---------- | ---------- |

## 7. 禁止事項

| 禁止事項                                                       | 理由                               |
| -------------------------------------------------------------- | ---------------------------------- |
| 品質目標を定性のみで記載する                                   | 判定不能になるため                 |
| メトリクスの算出方法を省略する                                 | 再現性が失われるため               |
| 是正プロセスを未定義にする                                     | 不適合が放置されるため             |
| 実装依存の詳細（SQL 全文、具体クラス名など）を本文に記載する   | 役割が混在するため                 |
| Agent に最終判断・公開可否判断を委ねる                         | 人間の判断責任を代替してしまうため |
| `pm-organization` で未採用の Role code を責任主体に使う        | 体制定義との不整合が生じるため     |
| 公開リポジトリに個人名・個人連絡先・非公開の組織情報を記載する | 公開範囲とプライバシーに反するため |
