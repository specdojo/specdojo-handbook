# 変更要求ログ 作成ルール

Change Request Log Documentation Rules

本ドキュメントは、`変更要求ログ` を一貫した粒度で作成するためのルールを定義する。

## 1. 全体方針

- 変更要求の申請から審査、決定、実施追跡までを判定可能な形で管理する。
- 変更要求ごとに識別子、影響評価、決定結果、責任者、期限を明記する。
- 曖昧表現を避け、承認可否や実施条件が判断できる記述にする。

## 2. 位置づけと用語定義（必要に応じて）

- `pm-plan`: プロジェクト全体の管理方針。
- `pm-issue-log`: 顕在化した課題の管理。
- `pm-risk-register`: 潜在リスクの管理。
- `pm-change-request-log`: 変更要求の審査・決定と実施追跡。

## 3. ファイル命名・ID規則

- `id` は `pm-change-request-log` を使用する。
- ファイル名は `pm-change-request-log-変更要求ログ.md` など、プロジェクト内で一意にする。

## 4. 推奨 Frontmatter 項目

| 項目 | 説明 | 必須 |
| --- | --- | --- |
| id | `pm-change-request-log` | ○ |
| type | `project` 固定 | ○ |
| status | `draft` / `ready` / `deprecated` | ○ |
| based_on | 根拠仕様ID（配列） | 任意 |
| supersedes | 置換関係ID（配列） | 任意 |

## 5. 本文構成（標準テンプレ）

| 番号 | 見出し | 必須 |
| --- | --- | --- |
| 1 | 概要 | ○ |
| 2 | 変更要求一覧 | ○ |
| 3 | 影響評価 | ○ |
| 4 | 審査・決定 | ○ |
| 5 | 実施追跡 | ○ |
| 6 | 関連ドキュメント | ○ |

## 6. 記述ガイド

### 6.1. 概要

- 管理対象、運用期間、記録単位を 1〜3 行で示す。

### 6.2. 変更要求一覧

- 変更 ID、要求内容、申請者、申請日、状態を一覧化する。

推奨フォーマット（表）:

| 変更ID | 要求内容 | 申請者 | 申請日 | 状態 |
| --- | --- | --- | --- | --- |

### 6.3. 影響評価

- 影響範囲（スコープ、工数、品質、納期）と評価根拠を明記する。

### 6.4. 審査・決定

- 審査結果、決定日、承認者、実施条件を明記する。

### 6.5. 実施追跡

- 実施担当、期限、進捗、完了条件を明記する。

### 6.6. 関連ドキュメント

- `pm-plan`、`pm-issue-log`、`pm-risk-register` との導線を記載する。
- 関連がない場合でも `（本書のみ）` を明記する。

## 7. 禁止事項

| 禁止事項 | 理由 |
| --- | --- |
| 変更IDを未採番で記録する | 追跡不能になるため |
| 影響評価なしで承認可否を記載する | 判断根拠が欠落するため |
| 決定者・決定日を省略する | 監査性が失われるため |
| 実装詳細やコード差分を本文に貼り付ける | 管理対象が肥大化するため |
