# 業務イベント仕様 全体構成 作成ルール

Business Event Specification Index (BES Index) Documentation Rules

業務で発生する主要イベント（販売、仕入、在庫調整など）を一覧化し、後続の BPS・BR・UI・テスト仕様と追跡可能につなぐための作成ルールを定義する。

## 1. 全体方針

- 業務イベントは「業務的に意味のある出来事」の一覧として整理し、実装設計ではなく概念レベルで記述する。
- 一覧は後工程（BPS / BR / UI / テスト）の参照元として扱い、イベントIDの一意性と可読性を優先する。
- 複雑なイベントは個別 BES へ分離し、一覧から相対リンクで辿れる構成を維持する。

## 2. 位置づけと用語定義

- 本ルールの対象は「業務イベント / 業務イベント仕様 全体構成（`bes-index`）」である。
- **イベントID** は一覧表の各行を一意に識別するIDで、`bes-<領域>-<短い英字>` を推奨する。
- **事実データ** はイベント発生時点で観測できる情報（ペイロード）の概念項目を指す。

## 3. ファイル命名・ID規則

### 3.1. ID規約

- 一覧ドキュメント自体の `id` は `bes-index` を推奨する。
- 一覧表のイベントIDは `^bes-[a-z0-9][a-z0-9-]*$` に合致する一意IDを用いる。
- 個別 BES ドキュメントを作成する場合、イベントIDとファイル名を対応させる（例: `bes-sale-checkout` ↔ `bes-sale-checkout.md`）。

### 3.2. ファイル命名規約

- 対象ドキュメントのファイル名は `bes-index.md` を推奨する。
- 個別イベント仕様のファイル名は `bes-<term>.md` を推奨する。
- 相対リンクで一覧から個別仕様へ遷移できる命名・配置を維持する。

## 4. 推奨 Frontmatter 項目

| 項目         | 説明                               | 必須 |
| ------------ | ---------------------------------- | ---- |
| `id`         | ドキュメントID（例: `bes-index`）  | ○    |
| `type`       | `project` を推奨（メタ仕様に従う） | ○    |
| `status`     | `draft` / `ready` / `deprecated`   | ○    |
| `rulebook`   | `bes-index-rulebook`               | 任意 |
| `supersedes` | 置き換え関係（古仕様→新仕様）      | 任意 |

## 5. 本文構成（標準テンプレ）

| 章  | 内容                   | 必須 |
| --- | ---------------------- | ---- |
| 1   | 業務イベント一覧（表） | ○    |

### 5.1. 業務イベント一覧の標準列

| 列名         | 説明                                  | 必須 |
| ------------ | ------------------------------------- | ---- |
| イベントID   | 一意識別子（例: `bes-sale-checkout`） | ○    |
| イベント名   | 業務ユーザーが理解可能な日本語名      | ○    |
| 何が起きたか | 業務的出来事を一文で記述              | ○    |
| いつ         | 発生タイミング・周期                  | ○    |
| 誰が／何が   | 人・システム・外部の起点              | ○    |
| 発生条件     | 観測条件・成立条件                    | ○    |
| 事実データ   | 観測可能情報（概念レベル）            | ○    |

### 5.2. 列順ルール

- 列順は次を固定する。
- イベントID → イベント名 → 何が起きたか → いつ → 誰が／何が → 発生条件 → 事実データ

## 6. 記述ガイド

### 6.1. 一覧記述の基本

- 一覧は表形式で記載する。
- 「何が起きたか」「いつ」「誰が／何が」「発生条件」は1セル1文の短文を基本とする。
- 事実データは実装カラム名ではなく業務概念名で列挙する。

### 6.2. リンクと責務分離

- 個別 BES を作成する場合、一覧のイベントIDを相対リンクとして記載する。
- イベントと業務プロセスは独立に管理し、起動関係は BPS 側で記述する。
- 用語は BPS / BR / UI / 用語集と整合する語彙を用いる。

### 6.3. ID・命名運用

- イベントIDは再利用せず、廃止時は置き換え関係を明示する。
- 略語は業務用語集で一般的に通用するもののみ使用する。
- 意味が推測しづらい内部コードはID・イベント名に使わない。

## 7. 禁止事項

- 物理テーブル名・物理カラム名・SQL全文を記載しない。
- 実装クラス名・関数名・API実装詳細を記載しない。
- UI操作手順（クリック列挙など）を一覧に混在させない。
- 曖昧語（その他、適宜など）を多用しない。
