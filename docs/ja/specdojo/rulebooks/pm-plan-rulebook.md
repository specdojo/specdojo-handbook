# プロジェクト管理計画 作成ルール

Project Management Plan Documentation Rules

本ドキュメントは、`プロジェクト管理計画` を一貫した粒度で作成するためのルールを定義する。

## 1. 全体方針

- プロジェクト全体の管理方針と運用プロセスを定義する。
- 管理領域（スコープ、スケジュール、コスト、品質、リスク、課題、変更管理）を一貫した粒度で記述する。
- 曖昧表現を避け、進行判断に使える記述にする。
- AI Agent は文書作成・レビュー・構成整理を支援するが、最終判断・公開可否判断は人間が行うことを明記する。
- 小規模プロジェクトでは `PO` が `PM` 責務を兼務する場合があることを許容し、その場合は本書に明記する。
- `based_on` には `prj-charter`、`pm-organization`、`pm-raci` を含めることを推奨する。

## 2. 位置づけと用語定義（必要に応じて）

- `pm-plan`: PM成果物群の中核となる計画書。
- `pm-risk-register`、`pm-issue-log`、`pm-change-request-log`: 監視・統制対象の実績管理。
- `pm-communication-plan`、`pm-organization`: 運営体制と連絡計画。

## 3. ファイル命名・ID規則

- ファイル名は `pm-plan.md` を推奨する。
- 推奨パス: `docs/ja/projects/<project-id>/030-project-management/010-management-plan/pm-plan.md`
- `id` は `<project-id>:pm-plan` 形式を推奨する。例: `prj-0001:pm-plan`

## 6. 記述ガイド

### 6.1. 概要

- 計画の目的、適用範囲、前提となる体制を 2〜4 行で示す。
- 本書が `prj-charter` の承認を前提とすること、および今後の見直し条件を概説する。
- AI Agent の役割（支援のみ、最終判断は人間）を明記する。

### 6.2. 管理方針（スコープ / スケジュール / コスト）

- 「2.1 基本方針」では管理領域ごとの管理単位・方針・判断者・逸脱時対応を表で示す。

推奨フォーマット:

| 管理領域 | 管理単位 | 方針 | 判断者 | 逸脱時対応 |
| -------- | -------- | ---- | ------ | ---------- |

- 「2.2 スコープ管理」では対象範囲の正本となる文書（`prj-charter`、`prj-scope`）と成果物カタログへの参照を記載する。
- 「2.3 スケジュール管理」では WBS → スケジュール YAML → 実行管理ビューの流れを示し、マイルストーン管理の正本を明示する。
- 「2.4 コスト管理」では金額予算がない場合でも、作業負荷・外部依存・運用負荷の観点と判断条件を記載する。

コスト管理の推奨フォーマット:

| 観点 | 管理内容 | 判断条件 |
| ---- | -------- | -------- |

### 6.3. 品質・リスク・課題管理方針

- 「3.1 品質管理」では本書の段階で適用する最低限の品質基準（構造、一貫性、利用可能性、公開適性）を表で示す。品質管理計画（`pm-quality-management-plan`）への委譲を明示する。
- 「3.2 リスク管理」ではリスク記録の正本（`pm-risk-register`）と記録トリガー条件を記載する。
- 「3.3 課題管理」では課題記録の正本（`pm-issue-log`）と優先順位判断者を記載する。
- 「3.4 変更管理」では変更要求として扱う対象と、軽微な修正との区別を記載する。

### 6.4. 体制と意思決定

- 「4.1 採用ロール」では `pm-organization` を正とし、本書の管理における各 Role code の扱いを記載する。
- 「4.2 意思決定」では判断対象・判断者・相談先・証跡を表で明示する。

推奨フォーマット:

| 判断対象 | 判断者 | 相談先 | 証跡 |
| -------- | ------ | ------ | ---- |

- 「4.3 RACI」では `pm-raci` を正とし、`pm-plan` 自身の責任分担のみ本書に記載する。

### 6.5. レポーティングと見直し

- 「5.1 レポーティング」では管理領域ごとの指標・報告頻度・判断者・閾値超過時対応を表で示す。

推奨フォーマット:

| 管理領域 | 指標 | 報告頻度 | 判断者 | 閾値超過時対応 |
| -------- | ---- | -------- | ------ | -------------- |

- 「5.2 見直し条件」では本書を更新するトリガーと見直し内容を表で示す。

推奨フォーマット:

| 更新トリガー | 見直し内容 |
| ------------ | ---------- |

- 「5.3 GO / Not GO 判断」では本格実行開始の判断基準を列挙し、判断先の文書（`prj-charter`）への参照を明示する。

### 6.6. 関連ドキュメント

- 各 PM 成果物（`pm-quality-management-plan`、`pm-communication-plan`、`pm-organization`、`pm-raci`、`pm-wbs-to-schedule-strategy`）への導線を記載する。
- プロジェクト定義文書（`prj-charter`、`prj-scope`、`prj-success-criteria-and-acceptance-criteria`）への参照を含める。

### 6.7. 継続確認事項（任意）

- 未決の論点、現状の暫定判断、今後の対応方針、判断者を表で管理する。

推奨フォーマット:

| 論点 | 現状 | 対応方針 | 判断者 |
| ---- | ---- | -------- | ------ |

## 7. 禁止事項

| 禁止事項                                              | 理由                               |
| ----------------------------------------------------- | ---------------------------------- |
| 管理領域を列挙のみで運用条件を書かない                | 実運用に使えないため               |
| 意思決定者を未記載にする                              | 判断遅延が発生するため             |
| 見直し条件を未定義にする                              | 計画が陳腐化するため               |
| 実装詳細（DB定義、API詳細等）を本文に記載する         | 役割が混在するため                 |
| Agent に最終判断・公開可否判断を委ねる                | 人間の判断責任を代替してしまうため |
| GO / Not GO の判断基準を定義しない                    | 本格実行開始の可否が不明になるため |
| `pm-organization` で未採用の Role code を判断者に使う | 体制定義との不整合が生じるため     |
