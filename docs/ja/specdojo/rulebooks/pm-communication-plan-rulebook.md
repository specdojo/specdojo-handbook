# コミュニケーション計画 作成ルール

Communication Plan Documentation Rules

本ドキュメントは、`コミュニケーション計画` を一貫した粒度で作成するためのルールを定義する。

## 1. 全体方針

- プロジェクトにおける報告、連絡、会議体、情報管理、エスカレーションの方針を計画として明文化する。
- コミュニケーション上の責任主体は Role code で記述し、`pm-organization` で採用済みの Role code のみを使う。
- 曖昧表現を避け、運用時に判断可能な記述にする。
- AI Agent は作業支援・レビュー支援を担ってよいが、最終判断や承認は人間の `PO` が行うことを明記する。
- 公開リポジトリに配置される場合は、個人名、個人連絡先、非公開の組織情報を記載しない。

## 2. 位置づけと用語定義

| ドキュメント            | 役割                                                                         |
| ----------------------- | ---------------------------------------------------------------------------- |
| `pm-plan`               | 全体管理方針。コミュニケーション計画は本書で別途定義する                     |
| `pm-organization`       | 採用ロールと未採用ロールを定義する。コミュニケーション上の責任主体の正とする |
| `pm-raci`               | 成果物・プロセスごとの責任分担を定義する                                     |
| `pm-communication-plan` | 報告・連絡・会議・情報管理・エスカレーションの実行計画                       |

## 3. ファイル命名・ID規則

- 推奨パス: `docs/ja/projects/<project-id>/030-project-management/010-management-plan/pm-communication-plan.md`
- `id` は `<project-id>:pm-communication-plan` 形式を推奨する。例: `prj-0001:pm-communication-plan`

## 6. 記述ガイド

### 6.1. 概要

- 計画の目的、適用範囲、前提とする体制（`pm-organization` 採用ロール）を 2〜4 行で示す。
- AI Agent の役割（支援のみ、最終判断は人間の `PO`）を明記する。
- 公開リポジトリの場合は、個人情報・機密情報を記載しない旨を明記する。

### 6.2. ロール別コミュニケーション分類

- `pm-organization` で採用済みの Role code を行として、コミュニケーション上の位置づけ・関心事項・受信情報・発信情報・頻度を記述する。
- 未採用 Role code の行は含めない。

推奨フォーマット:

| Role code | コミュニケーション上の位置づけ | 主な関心事項 | 受信情報 | 発信情報 | 頻度 / タイミング |
| --------- | ------------------------------ | ------------ | -------- | -------- | ----------------- |

### 6.3. 会議体計画

- 会議名、目的、参加ロール（Role code）、頻度、成果物/証跡を明記する。
- 個人・小規模運用では定例会議を省略してよい。省略する場合は証跡の代替手段を明示する。

推奨フォーマット:

| 会議名 | 目的 | 参加ロール | 頻度 | 成果物 / 証跡 |
| ------ | ---- | ---------- | ---- | ------------- |

### 6.4. 報告計画

- 「4.1 報告種別」では報告種別・受信ロール・目的・期限/タイミング・形式・責任ロールを表で示す。

推奨フォーマット:

| 報告種別 | 受信ロール | 目的 | 期限 / タイミング | 形式 | 責任ロール |
| -------- | ---------- | ---- | ----------------- | ---- | ---------- |

- 「4.2 報告に含める内容」では進捗報告・作業節目で共有すべき事項を箇条書きで示す。

### 6.5. 連絡チャネルとエスカレーション

- 「5.1 連絡チャネル」ではチャネル・用途・使用条件・証跡を表で示す。

推奨フォーマット:

| チャネル | 用途 | 使用条件 | 証跡 |
| -------- | ---- | -------- | ---- |

- 「5.2 エスカレーション」では区分・一次対応・エスカレーション条件・最終判断・証跡を表で示す。

推奨フォーマット:

| 区分 | 一次対応 | エスカレーション条件 | 最終判断 | 証跡 |
| ---- | -------- | -------------------- | -------- | ---- |

### 6.6. 情報管理と公開方針

- 「6.1 証跡管理」で重要判断の証跡残し方針と、生成物と正本が矛盾した場合の扱いを記載する。
- 「6.2 公開情報の扱い」で公開リポジトリに含めない情報の種類と管理方針を記載する。

### 6.7. 見直し条件

- 本書を更新するトリガーと見直し内容、責任ロールを表で示す。

推奨フォーマット:

| 更新トリガー | 見直し内容 | 責任ロール |
| ------------ | ---------- | ---------- |

## 7. 禁止事項

| 禁止事項                                                                  | 理由                               |
| ------------------------------------------------------------------------- | ---------------------------------- |
| 連絡先や責任者を未記載にする                                              | 伝達遅延が発生するため             |
| 会議体の目的や成果物/証跡を未定義にする                                   | 会議が形骸化するため               |
| エスカレーション条件を曖昧にする                                          | 障害時判断が遅れるため             |
| 実装詳細を本文に含める                                                    | ドキュメント責務から逸脱するため   |
| Agent に最終判断・承認を委ねる                                            | 人間の判断責任を代替してしまうため |
| `pm-organization` で未採用の Role code をコミュニケーション責任主体に使う | 体制定義との不整合が生じるため     |
| 公開リポジトリに個人名・個人連絡先・非公開の組織情報を記載する            | 公開範囲とプライバシーに反するため |
