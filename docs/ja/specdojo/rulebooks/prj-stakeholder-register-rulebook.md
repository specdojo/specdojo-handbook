# ステークホルダー登録簿 作成ルール

Stakeholder Register Documentation Rulebook

本ドキュメントは、プロジェクトの関係者を識別し、影響度と関与方針を可視化するためのルールです。
関係者との合意形成とコミュニケーション統制に使える粒度を定義します。

## 1. 全体方針

- 本ルールの対象は、ステークホルダー登録簿（関係者の識別、分析、対応方針）です。
- 目的は、意思決定者・影響者・実行者・利用者・影響を受ける関係者を明確化し、合意形成の遅延リスクを抑えることです。
- プロジェクト概要/憲章で定義した背景と認可情報を入力として扱い、本書で重複管理しません。
- ステークホルダーに対応する Role code がある場合は、対応 Role code を明示し、組織定義、RACI、WBS / Schedule へ接続できるようにします。
- PMBOK 観点では、ステークホルダー特定、エンゲージメント計画、監視更新に必要な情報を含めます。
- コミュニケーション計画の入力情報として、関係者ごとの情報要求、関与方針、合意事項を整理します。
- 曖昧語を避け、更新条件、責任者、証跡を記載します。

## 2. 位置づけと用語定義

### 2.1. 位置づけ（他ドキュメントとの関係）

ドキュメント間の関係は [ドキュメント構成ガイド](../guidelines/docs-structure-guide.md) の立ち上げフロー図を参照してください。本書（`prj-stakeholder-register`）は `prj-overview` を受けて関係者を特定し、`prj-charter` と `pm-organization` の入力となります。

### 2.2. 用語定義（本ルール内）

| 用語             | 定義                                                          |
| ---------------- | ------------------------------------------------------------- |
| ステークホルダー | プロジェクトの成果に影響を与える、または影響を受ける個人/組織 |
| 関与区分         | ステークホルダーがプロジェクトに関わる立場や役割の分類        |
| 影響度           | 意思決定や実行に与える影響の強さ                              |
| 関心度           | プロジェクト成果に対する関与意欲・関心の高さ                  |
| エンゲージメント | 関係者が期待される状態で関与している度合い                    |
| 必要な合意       | プロジェクトを進めるために当該関係者と確認・承認すべき事項    |
| 対応方針         | 関係者ごとに定義する情報提供・調整・合意の進め方              |
| 対応 Role code   | ステークホルダーの期待、懸念、合意対象を主に扱う Role code    |

## 3. ファイル命名・ID規則

### 3.1. 配置（推奨）

- `docs/ja/projects/<project-id>/020-project-definition/` 配下への配置を推奨します。
- 関係者ごとの補足資料（議事録、合意記録、依頼メモ）は同階層または関連する管理文書配下に配置し、登録簿から参照します。
- 公開リポジトリに配置する場合は、個人情報、非公開の組織情報、機密性の高い利害関係を記載しません。

### 3.2. ドキュメントID（推奨）

- 推奨: `<project-id>-prj-stakeholder-register`
  - 例: `prj-0001-prj-stakeholder-register`

### 3.3. ファイル名（推奨）

- 推奨: `prj-stakeholder-register.md`
- 日本語ファイル名の場合: `ステークホルダー登録簿.md`

## 4. 推奨 Frontmatter 項目

### 4.1. 設定内容

- 参照スキーマ: [docs/specdojo/schemas/v1/deliverable-frontmatter.schema.yaml](../../../specdojo/schemas/v1/deliverable-frontmatter.schema.yaml)
- メタ情報標準: [deliverable-metadata-standard.md](../standards/deliverable-metadata-standard.md)

| 項目       | 説明                                    | 必須 |
| ---------- | --------------------------------------- | ---- |
| id         | `<project-id>-prj-stakeholder-register` | ○    |
| type       | `project` 固定                          | ○    |
| status     | `draft` / `ready` / `deprecated`        | ○    |
| rulebook   | `prj-stakeholder-register-rulebook`     | ○    |
| based_on   | 憲章、組織定義、ロール定義、既存連絡網  | 任意 |
| supersedes | 置き換え対象の旧文書 ID                 | 任意 |

### 4.2. 推奨ルール

- 個人情報を含む場合は記載範囲を最小化し、必要時はアクセス制限された参照先へ分離します。
- 対応 Role code を記載する場合は、`pm-roles.yaml` または `pm-organization.md` で採用済みの Role code だけを使用します。
- 個人名の記載が不要な場合は、組織名、役割名、窓口名で代替します。
- メールアドレス、電話番号、住所等の連絡先情報は、必要最小限に留めます。
- 公開リポジトリに配置する文書には、個人情報、非公開の組織情報、機密性の高い利害関係を記載しません。
- 改訂時は、変更理由と変更対象（追加/削除/役割変更）を記録します。

## 6. 記述ガイド

### 6.1. 共通

- 1 ステークホルダーにつき、関与区分、責任、期待、懸念、リスク、必要な合意、対応方針を 1 セットで記載します。
- 登録対象は、プロジェクトの意思決定、実行、利用、運用、評価、資金提供、影響受容のいずれかに関係する個人・組織とします。
- 直接関与者だけでなく、成果物の利用者、運用担当者、承認者、外部協力者、影響を受ける外部関係者も必要に応じて含めます。
- 対応する Role code がある場合は、ステークホルダーと Role を混同しないよう、対応 Role code として分けて記載します。
- 対応 Role code がない外部基盤や外部集団には、無理に Role code を割り当てません。
- 関係者の分類は固定語彙を使います。
  - 例: スポンサー / 意思決定者 / 実行責任者 / 実行担当者 / 利用者 / 運用担当者 / 外部協力者 / 影響受容者

- 章参照は章番号ではなく章タイトルで記載します。

### 6.2. 関係者一覧

- 関係者一覧では、ステークホルダーを識別し、プロジェクトにおける立場と責任を明確にします。
- ID は People and Organization Definition Standard のステークホルダー ID 規則に従い、`STH-<KEY>` 形式の意味付きキーを使用します。
- `対応 Role code` には、当該ステークホルダーの期待、懸念、合意対象を主に扱う Role code を記載します。
- 1 つのステークホルダーに複数の Role code が対応してよいです。
- 対応 Role code は、`pm-roles.yaml` または `pm-organization.md` で採用済みの Role code に限定します。
- 個人名が不要な場合は、組織名、役割名、窓口名で記載します。
- 公開文書では、個人名や連絡先を原則として記載せず、必要に応じて非公開の連絡先管理資料へ分離します。

推奨表（関係者一覧）:

| ID | 関係者 | 関与区分 | 所属/組織 | 対応 Role code | 主な責任 | 備考 |
| -- | ------ | -------- | --------- | -------------- | -------- | ---- |

### 6.3. 影響度/関心度分析

- 影響度と関心度は High/Medium/Low 等で定義し、評価基準を明示します。
- 評価は主観だけで行わず、意思決定権限、予算権限、利用影響、運用責任、合意形成への影響などの根拠を記載します。
- 高影響・低関心の関係者は、巻き込み施策を別途定義します。
- 高関心・低影響の関係者は、過剰な負荷をかけず、適切な情報共有方法を定義します。

標準評価基準:

| 評価   | 影響度の基準                                     | 関心度の基準                         |
| ------ | ------------------------------------------------ | ------------------------------------ |
| High   | 予算、方針、スコープ、リリース可否に直接影響する | 成果・進行状況に継続的な関与を求める |
| Medium | 一部の成果物、運用、利用、合意形成に影響する     | 必要時に確認・調整を行う             |
| Low    | 影響は限定的または間接的である                   | 通知・共有のみで足りる               |

推奨表（関係者分析）:

| ID | 関係者 | 関与区分 | 対応 Role code | 影響度 | 関心度 | 主な期待 | 主な懸念 | 必要な合意 | 評価根拠 |
| -- | ------ | -------- | -------------- | ------ | ------ | -------- | -------- | ---------- | -------- |

### 6.4. エンゲージメント方針

- 現状と目標のエンゲージメント状態を比較し、差分対策を記載します。
- 対応方針には責任者と実施期限を含めます。
- 対応方針は、情報提供、相談、合意形成、承認依頼、巻き込み、リスク低減のいずれを目的とするかが分かるように記載します。
- 証跡には、議事録、Issue、Pull Request、合意記録、メール、チャット、決定記録などを参照します。

推奨表（対応方針）:

| ID  | 関係者 | 現状 | 目標 | 対応方針 | 責任者 | 期限 | 証跡 |
| --- | ------ | ---- | ---- | -------- | ------ | ---- | ---- |

### 6.5. コミュニケーション要件

- ステークホルダー登録簿では、関係者ごとのコミュニケーション要件を定義します。
- 会議体、報告頻度、配信先、議事録管理等の詳細はコミュニケーション計画で管理します。
- コミュニケーション要件には、情報要求、希望チャネル、合意・報告の必要性、証跡要件を含めます。
- コミュニケーション計画を作成する場合は、本章の内容を入力として扱います。

推奨表（コミュニケーション要件）:

| ID  | 関係者 | 情報要求 | 希望チャネル | 合意・報告の必要性 | 証跡要件 | コミュニケーション計画への反映 |
| --- | ------ | -------- | ------------ | ------------------ | -------- | ------------------------------ |

### 6.6. 見直し条件

- ステークホルダー登録簿は、プロジェクト開始時だけでなく、主要な変更時に見直します。
- 見直し条件には、更新トリガー、見直し周期、承認者を記載します。
- 変更履歴は、文書管理ルールまたは Git 履歴で追跡します。必要に応じて本文内にも要約を記載します。

推奨表（見直し条件）:

| 更新トリガー           | 見直し内容                               | 見直し責任者 | 承認者 | 証跡 |
| ---------------------- | ---------------------------------------- | ------------ | ------ | ---- |
| プロジェクト憲章の変更 | 関係者、影響度、合意事項の再確認         |              |        |      |
| スコープ変更           | 影響を受ける関係者と必要な合意の再確認   |              |        |      |
| 体制変更               | 役割、責任、連絡経路の更新               |              |        |      |
| リリース前             | 承認者、利用者、運用担当者の合意状況確認 |              |        |      |
| 運用開始後の主要変更   | 利用者、運用担当者、影響受容者の再確認   |              |        |      |

## 7. 禁止事項

| 項目                               | 理由                                             |
| ---------------------------------- | ------------------------------------------------ |
| 役割や責任の未記載                 | 調整漏れと責任不明確を招くため                   |
| 登録対象の識別基準なし             | 重要な関係者の漏れを招くため                     |
| ステークホルダー ID を WBS / Schedule の `owner` に使う | Stakeholder と Role の責務境界が崩れるため |
| 未採用 Role code を対応 Role code に記載する | 組織定義、RACI、WBS / Schedule との不整合を招くため |
| 対応 Role code がない対象へ無理に Role code を割り当てる | 外部基盤や外部集団の扱いを誤るため |
| 影響度/関心度の根拠なし評価        | 対応優先度を誤るため                             |
| 必要な合意の未記載                 | 合意形成の遅延や承認漏れを招くため               |
| 個人情報の過剰記載                 | 情報管理リスクを高めるため                       |
| 公開文書への機密情報の記載         | 個人情報・非公開情報・利害関係の漏えいを招くため |
| 関係者更新の未記録                 | 合意形成プロセスの追跡ができないため             |
| コミュニケーション計画との重複管理 | 文書間の不整合と更新漏れを招くため               |
