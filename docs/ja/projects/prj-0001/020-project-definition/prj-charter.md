# プロジェクト憲章: SpecDojo Handbook

## 1. 本書の目的

本書は、SpecDojo Handbook プロジェクトの立ち上げを正式に承認し、プロジェクトマネージャーに詳細計画の策定および初期準備を進める権限を付与するための文書である。

プロジェクトの背景、目的、必要性、期待効果、前提条件の詳細は [prj-overview.md](./prj-overview.md) を正とする。

本書では、プロジェクト立ち上げの承認に必要な高レベルの要点と、初期段階における権限委譲の範囲を定める。

なお、本書の承認は本格実行の開始を意味しない。
本格実行の開始可否は、プロジェクトスコープ、成功基準、成果物カタログ、前提・制約・依存関係、およびプロジェクト管理計画の作成後に、GO / Not GO 判断として別途確認する。

## 2. 認可対象

本書により、次のプロジェクトの立ち上げを正式に認可する。

| 項目                 | 内容                                                               |
| -------------------- | ------------------------------------------------------------------ |
| プロジェクト名       | SpecDojo Handbook                                                  |
| プロジェクトID       | prj-0001                                                           |
| 認可対象             | SpecDojo Handbook プロジェクトの立ち上げ、詳細計画の策定、初期準備 |
| プロジェクト概要     | [prj-overview.md](./prj-overview.md)                               |
| 初期ステークホルダー | [prj-stakeholder-register.md](./prj-stakeholder-register.md)       |

本憲章の承認により、プロジェクトマネージャーは、詳細なスコープ、成果物、成功基準、管理方針、体制、スケジュールを定義するための作業に着手できる。

## 3. プロジェクトの目的

本プロジェクトの目的は、仕様駆動開発のための、ライフサイクル全体を支えるオープンなドキュメントフレームワーク **SpecDojo** の **Handbook** を整備することである。

SpecDojo Handbook では、システムの構築・改修・運用・保守に必要な文書群を体系化し、記述規約、テンプレート、サンプル、運用ルール、AI 向け指示テンプレート、補助ツールを通じて、企画から継続改善までを一貫して支援する。

本プロジェクトは、人と AI が共有可能な仕様を中核に据えることで、開発の生産性、品質、継続性を抜本的に高めることを目指す。

また、SpecDojo Handbook をオープンソースとして公開し、仕様駆動開発の知見を、誰もが自由に利用・改善・再利用できる共通基盤として提供する。

詳細は [prj-overview.md](./prj-overview.md) を正とする。

## 4. ハイレベルスコープ

本憲章では、次の範囲を初期検討および詳細計画策定の対象として認可する。

- SpecDojo Handbook の文書体系の定義
- プロジェクト文書、プロダクト文書、運用文書等の分類と構成の整理
- 文書記述ルールの整備
- テンプレート、サンプル、AI 向け指示テンプレートの整備
- `specdojo` コマンド等の補助ツールの整備
- オープンソースとして公開・改善・再利用しやすい運用基盤の整備

詳細な対象範囲、対象外、境界条件は、本憲章の承認後に作成する [prj-scope.md](./prj-scope.md) で定義する。

## 5. ハイレベル成果物

本プロジェクトでは、現時点で次の成果物群を整備対象の候補とする。

- プロジェクト定義文書
- プロジェクトスコープ関連文書
- 課題・解決アプローチ関連文書
- プロダクト変更関連文書
- プロジェクトマネジメント関連文書
- 決定記録
- 文書ルール
- テンプレート
- サンプル
- AI 向け指示テンプレート
- 補助ツールおよび生成成果物

成果物の詳細な一覧、配置、生成元、派生関係は、本憲章の承認後に作成する [prj-deliverables-catalog.md](../010-deliverables-catalog/prj-deliverables-catalog.md) で定義する。

## 6. ハイレベル成功基準

本プロジェクトでは、現時点で次の観点を成功判定の候補とする。

- 仕様駆動開発に必要な主要文書体系が整理されていること
- 各文書の作成ルール、テンプレート、サンプルが整備されていること
- 人と AI が参照しやすい構造で文書が管理されていること
- 変更時に関連文書を追跡・更新できる考え方が整理されていること
- オープンソースとして公開・利用・改善・再利用しやすい構成になっていること

具体的な成功指標、目標値、受入条件、判定方法、承認者は、本憲章の承認後に作成する [prj-success-criteria-and-acceptance-criteria.md](./prj-success-criteria-and-acceptance-criteria.md) で定義する。

## 7. 初期ステークホルダー

初期ステークホルダーは [prj-stakeholder-register.md](./prj-stakeholder-register.md) を正とする。

本憲章では、プロジェクト立ち上げ時点の主要ステークホルダーを次のとおり定める。

| 役割                     | 関与区分                             | 主な責任                                                             |
| ------------------------ | ------------------------------------ | -------------------------------------------------------------------- |
| プロジェクトオーナー     | スポンサー / 意思決定者 / 実行責任者 | プロジェクトの目的、方針、優先順位、公開方針を決定する               |
| プロジェクトマネージャー | 実行責任者                           | 詳細計画の策定、初期準備、成果物作成、課題・リスク・変更の管理を行う |
| AI Agent                 | 実行支援者                           | 文書作成、レビュー、構成整理、改善提案を支援する                     |
| 将来の利用者             | 利用者                               | SpecDojo Handbook を参照・利用し、仕様駆動開発を実践する             |
| 将来の貢献者             | 外部協力者 / 新規貢献者              | Issue、Pull Request、提案等を通じて改善に参加する                    |

現時点では個人プロジェクトとして開始するため、プロジェクトオーナーとプロジェクトマネージャーは同一主体が兼ねることを許容する。

## 8. 権限委譲

本憲章の承認により、プロジェクトマネージャーは、次の権限を持つ。

- 詳細なプロジェクトスコープを作成する
- 成功基準および受入条件を整理する
- 成果物カタログを作成する
- 前提・制約・依存関係を整理する
- プロジェクト管理計画を作成する
- 組織体制と RACI を整理する
- 作成対象文書の優先順位を調整する
- AI Agent を活用して文書の草案作成、レビュー、改善を進める
- 初期成果物の作成、更新、レビューを進める
- リポジトリ構成、README、ライセンス、貢献導線の整備を検討する

ただし、次の事項はスポンサーまたはプロジェクトオーナーの承認を必要とする。

- プロジェクト目的の変更
- ハイレベルスコープの大幅な変更
- 主要成果物群の追加・削除
- オープンソース公開方針の変更
- ライセンス方針の変更
- 本格実行開始の GO / Not GO 判断
- プロジェクト中止または大幅な再計画の判断

## 9. 主要前提・制約

本憲章の承認時点では、次の前提・制約を置く。

| 区分 | 内容                                                                                                           |
| ---- | -------------------------------------------------------------------------------------------------------------- |
| 前提 | SpecDojo Handbook は、人間の判断や責任を代替するものではなく、人と AI が協働するための基盤として整備する       |
| 前提 | 対象範囲は、企画、要件定義、設計、実装、テスト、運用・保守、継続改善を含むシステムライフサイクル全体を想定する |
| 前提 | SpecDojo Handbook はオープンソースとして公開し、自由に利用・改善・再利用できる共通基盤とする                   |
| 前提 | 初期段階では個人プロジェクトとして開始し、必要に応じて外部利用者・貢献者を受け入れる                           |
| 制約 | 特定の開発手法、言語、基盤、生成 AI 製品に過度に依存しない                                                     |
| 制約 | 公開リポジトリには、個人情報、非公開の組織情報、機密性の高い利害関係を記載しない                               |
| 制約 | 詳細な管理方針、スケジュール、体制、リスク、コミュニケーション方針は、本憲章承認後に作成する関連文書で定義する |

詳細な前提、制約、依存関係は、本憲章の承認後に作成する [prj-assumptions-constraints-dependencies.md](./prj-assumptions-constraints-dependencies.md) で定義する。

## 11. 本格実行開始の GO / Not GO 判断

本憲章の承認後、プロジェクトは詳細計画策定および初期準備の段階へ移行する。

本格実行を開始する前に、プロジェクト管理計画の承認時点で GO / Not GO 判断を行う。

GO / Not GO 判断では、少なくとも次の観点を確認する。

| 判定観点   | 確認内容                                                                         | 主な参照文書                                          |
| ---------- | -------------------------------------------------------------------------------- | ----------------------------------------------------- |
| スコープ   | 対象範囲と対象外が定義されていること                                             | `prj-scope`                                           |
| 成功基準   | 成功基準、受入条件、判定方法が定義されていること                                 | `prj-success-criteria-and-acceptance-criteria`        |
| 成果物     | 主要成果物と管理方法が定義されていること                                         | `prj-deliverables-catalog`                            |
| 前提・制約 | 実行上の前提、制約、依存関係が整理されていること                                 | `prj-assumptions-constraints-dependencies`            |
| 体制       | 役割と責任分担が定義されていること                                               | `pm-organization-and-raci`                            |
| 管理方針   | 進捗、品質、リスク、課題、変更、コミュニケーションの管理方針が定義されていること | `pm-plan`                                             |
| 公開方針   | オープンソース公開、ライセンス、個人情報・機密情報の扱いが確認されていること     | `pm-plan`, `prj-assumptions-constraints-dependencies` |

GO / Not GO 判断の結果は、[pm-plan.md](../030-project-management/010-management-plan/pm-plan.md) または関連する決定記録に記録する。
