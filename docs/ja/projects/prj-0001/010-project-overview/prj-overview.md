# プロジェクト概要: SpecDojo Handbook

## 1. 背景

生成AIおよび各種Agentの発展により、システムの企画、要件定義、設計、実装、テスト、運用・保守に至るまで、自動化・高度化可能な業務領域が急速に拡大している。

これにより、従来は専門人材や大規模な開発体制を必要としていたシステム構築・改修の一部を、より小さなチーム、非専門家を含む組織、オープンソースコミュニティ、地域・教育・福祉等の現場でも実行できる可能性が広がっている。

一方で、AIが適切にタスクを実行するためには、要件、仕様、設計、制約条件、運用方針、意思決定の経緯等が、曖昧さなく、構造的に、継続的に記述・共有されている必要がある。

従来、開発文書は主として人間同士の理解と合意形成のためのものであった。しかし今後は、人間に加えてAIも読み手・実行主体となることから、ドキュメントには可読性だけでなく、解釈可能性、再利用性、追跡可能性、実行可能性が求められる。

本プロジェクトでは、このようなドキュメントを接点として、人とAIが協働しながらシステムを構築・改修・運用し、継続的に改善していく開発スタイルを**仕様駆動開発**と位置付ける。

## 2. 目的

仕様駆動開発のための、オープンかつライフサイクル全体を対象としたドキュメントフレームワーク **SpecDojo** を整備する。

SpecDojo では、システムの構築・改修・運用・保守に必要な文書群を体系化し、記述規約、テンプレート、サンプル、運用ルール、AI向け指示テンプレート、補助ツールを通じて、企画から運用・保守、継続改善までを一貫して支援する。

本プロジェクトの目的は、単に開発文書の形式を標準化することではない。
人とAIが共有可能な仕様を中核に据えることで、開発活動の抜本的な生産性向上、品質向上、属人化の抑制、知識継承、継続的改善を実現し、持続可能なシステム開発・運用の基盤を確立することである。

あわせて、以下を満たすドキュメンテーションの実践基盤を確立する。

- AIが構造的に解釈し、タスク実行に活用しやすいこと
- 人が理解し、判断し、合意形成しやすいこと
- 変更理由、設計意図、制約条件を追跡できること
- 継続的に記述・更新・再利用できること
- 特定の組織、製品、開発手法に閉じず、幅広い現場で適用できること

これにより、企業、行政、教育、医療、福祉、地域活動、オープンソース等の多様なプロジェクトにおいて、限られた人材・時間・予算の中でも、社会課題の解消に資するシステムを継続的に構築・改善できる状態を目指す。

## 3. 必要性

システム開発におけるドキュメンテーションは、これまで担当者の経験や力量に依存しやすく、標準化・再利用・継続的更新が進みにくかった。

その結果、要件や設計意図の伝達不足による認識齟齬、手戻り、属人化、ブラックボックス化、保守困難化が多くのプロジェクトで課題となっている。

特に、社会課題の解消に取り組むプロジェクトでは、限られたリソースの中で、継続的にシステムを改善し続ける必要がある。しかし、仕様や設計意図が十分に残されていない場合、担当者の交代、予算の制約、技術変化、運用環境の変化に対応できず、価値ある取り組みが継続困難になることがある。

さらに、AIが実装、テスト、レビュー、運用支援等の作業を担うようになったことで、ドキュメントには従来の人間向け記述に加え、AIが構造的に解釈し、具体的なタスク実行に利用できることが求められている。

しかし、どの文書を、どの粒度で、どのような構造・規約で整備すべきかについては、現時点で共通基盤が十分に存在しない。
また、AI活用を前提としたドキュメント体系は、個別のプロンプト作成や単発の自動生成にとどまりやすく、ライフサイクル全体を支える継続的な開発・運用基盤としては未成熟である。

このため、仕様駆動開発に適したドキュメンテーションを再定義し、体系化し、現場で継続運用可能な形で提示することが必要である。

SpecDojo は、開発文書を単なる成果物ではなく、人とAIが協働するための共有知識基盤として位置付ける。
これにより、個別プロジェクトの効率化にとどまらず、多様な主体による持続可能な開発活動を支え、社会課題の解消を加速することを目指す。

## 4. 期待効果

SpecDojo Handbook の整備により、以下の効果を期待する。

- 要求・仕様・設計に関する認識齟齬の削減
- 開発初期からの手戻り抑制
- AI Agent に対する入力品質向上による成果物品質の安定化
- ドキュメント構成・記述ルールの共通化による知見の再利用促進
- 属人化・ブラックボックス化の抑制
- 新規参画者の早期立ち上がり
- 変更理由・設計意図・制約条件の追跡性向上
- 保守・改修・引き継ぎに要する負荷の低減
- 小規模チームや非専門家を含む現場における開発実行力の向上
- オープンソースプロジェクトにおける参加・貢献・継続改善の促進
- ライフサイクル全体での生産性、品質、継続性の向上

中長期的には、以下の効果を目指す。

- 限られた人材・予算でも、価値あるシステムを継続的に構築・改善できること
- 企業、行政、教育、医療、福祉、地域活動、OSS 等の多様な領域で、課題解決型プロジェクトの立ち上げと継続が容易になること
- AI活用が一部の高度人材や大規模組織に偏らず、より多くの現場で実践可能になること
- 仕様を中心とした知識共有により、プロジェクト間での知見再利用が進むこと
- 結果として、社会課題の解消に向けたシステム開発・改善の速度と持続性が高まること

可能な範囲で、以下のような指標により効果を測定する。

| 指標                                        | 現状   | 目標     | 測定方法                                       |
| ------------------------------------------- | ------ | -------- | ---------------------------------------------- |
| 要件定義〜基本設計における手戻り件数        | 未測定 | 30%削減  | 変更管理票・課題管理票の比較                   |
| AIによる実装タスクの再実行率                | 未測定 | 20%削減  | 再指示・再生成回数の記録                       |
| AI生成成果物のレビュー指摘件数              | 未測定 | 25%削減  | レビュー指摘件数の集計                         |
| 新規参画者の立ち上がり期間                  | 未測定 | 50%短縮  | 初回成果物提出までの日数を計測                 |
| 主要ドキュメントの整備率                    | 未測定 | 90%以上  | 必須文書の整備状況を確認                       |
| 変更時の関連文書更新率                      | 未測定 | 90%以上  | 変更時の文書更新状況を確認                     |
| 仕様・設計意図の追跡可能項目率              | 未測定 | 80%以上  | 要件、設計、実装、テスト間の参照関係を確認     |
| 保守・改修時の調査時間                      | 未測定 | 30%削減  | 改修着手までの調査時間を記録                   |
| OSS等における新規貢献者の初回貢献までの期間 | 未測定 | 30%短縮  | 初回Issue対応・PR作成までの日数を計測          |
| ドキュメント再利用件数                      | 未測定 | 継続増加 | テンプレート、サンプル、ルールの利用実績を記録 |

なお、社会課題の解消に対する貢献は、単一の短期指標だけで測定することは難しい。
そのため、短期的には開発生産性・品質・継続性に関する指標を測定し、中長期的には、SpecDojo を活用したプロジェクト数、継続率、OSS への貢献状況、社会課題領域での活用事例等を通じて効果を確認する。

## 5. 前提条件

SpecDojo Handbook の策定および活用にあたっては、以下を前提とする。

- SpecDojo は、人間の判断や責任を代替するものではなく、人とAIが協働するための基盤であること
- 対象範囲は、企画、要件定義、設計、実装、テスト、運用・保守、継続改善を含むシステムライフサイクル全体とすること
- ドキュメントは、人間にとっての理解容易性と、AIにとっての解釈容易性を両立すること
- ドキュメントは、成果物であると同時に、判断、合意、実行、改善のための共有知識基盤であること
- 記述ルールは、現場で継続運用可能な粒度・負荷に収めること
- Handbook は文書規約にとどまらず、テンプレート、サンプル、運用ルール、補助ツール、AI向け指示テンプレート等を含むこと
- 特定の開発手法、言語、基盤、生成AI製品に過度に依存しないこと
- 企業、行政、教育、医療、福祉、地域活動、オープンソース等、多様な主体が利用可能なオープンな基盤を目指すこと
- 社会課題の解消という長期的な目的を見据えつつ、まずは実践可能な文書体系と運用ルールから段階的に整備すること
