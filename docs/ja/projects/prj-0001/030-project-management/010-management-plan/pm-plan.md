# プロジェクト管理計画

## 1. 概要

本書は、SpecDojo プロジェクトの管理方針、運用プロセス、意思決定、監視・見直し方法を定義する。

本プロジェクトは個人・小規模運用を前提とし、プロジェクトオーナーがプロジェクトマネージャー相当の責務を兼務する。AI Agent は文書作成、レビュー、構成整理、改善提案を支援するが、最終判断や公開可否判断は人間が行う。

本書はドラフトであり、プロジェクトスコープ、成功基準、成果物カタログ、前提・制約・依存関係、品質管理計画、コミュニケーション計画の更新に応じて見直す。

## 2. 管理方針（スコープ / スケジュール / コスト）

### 2.1. 基本方針

| 管理領域 | 管理単位 | 方針 | 判断者 | 逸脱時対応 |
| -------- | -------- | ---- | ------ | ---------- |
| スコープ | 成果物カタログ、WBS、プロジェクト登録簿 | 対象成果物と作業範囲は成果物カタログと WBS で管理する | `PO` | 追加・削除・大幅変更はプロジェクト登録簿へ記録し、必要に応じて変更要求として扱う |
| スケジュール | マイルストーン、スケジュール YAML、実行管理ビュー | WBS からスケジュールを作成し、進捗と遅延を確認する | `PO` | 重要タスクの遅延、依存関係の破綻、マイルストーン影響がある場合は計画を見直す |
| コスト | 作業時間、外部サービス利用、公開・運用負荷 | 初期段階では詳細な金額管理ではなく、作業負荷と外部依存の増加を管理する | `PO` | 有償サービス、外部委託、継続費用が必要になった場合は事前に判断する |
| 成果物 | work、control、generated の各成果物 | work 成果物を正とし、control / generated は運用・監視の補助として扱う | `PO` | 正となる文書と生成物の差異がある場合は、正本を更新してから再生成する |

### 2.2. スコープ管理

- プロジェクトの対象範囲は [prj-charter.md](../../020-project-definition/prj-charter.md) と [prj-scope.md](../../020-project-definition/prj-scope.md) を正とする。
- 成果物の一覧、配置、種別、派生関係は [dct-index.md](../../010-deliverables-catalog/dct-index.md) およびドメイン別成果物カタログで管理する。
- 新しい成果物を追加する場合は、成果物カタログ、必要に応じて WBS / Schedule / RACI を更新する。
- 対象外、未決、後続判断が必要な事項は、プロジェクト登録簿または該当する定義文書に明示する。

### 2.3. スケジュール管理

- 作業は WBS YAML を起点に分解し、スケジュール YAML と実行管理ビューで管理する。
- マイルストーンは [sch-milestones.yaml](../schedule/sch-milestones.yaml) を正とする。
- WBS からスケジュールへの展開方針は [pm-wbs-to-schedule-strategy.md](pm-wbs-to-schedule-strategy.md) を正とする。
- スケジュール差分、依存関係、クリティカルパスは、必要に応じて `execution/generated` 配下の生成物で確認する。

### 2.4. コスト管理

本プロジェクトでは、初期段階の金額予算を設定しない。代わりに、次の観点をコスト相当の制約として管理する。

| 観点 | 管理内容 | 判断条件 |
| ---- | -------- | -------- |
| 作業負荷 | 文書作成、レビュー、再生成、整合性確認に必要な時間 | 継続更新が困難な粒度・量になった場合 |
| 外部依存 | GitHub、生成 AI、公開基盤、補助ツールへの依存 | 有償化、利用制限、代替困難性が発生した場合 |
| 運用負荷 | Issue / PR、公開後の問い合わせ、更新頻度 | 個人運用で対応困難な量になった場合 |

## 3. 品質・リスク・課題管理方針

### 3.1. 品質管理

- 品質管理の詳細は [pm-quality-management-plan.md](pm-quality-management-plan.md) で定義する。
- 本計画の段階では、最低限の品質基準として、次を適用する。

| 観点 | 基準 | 確認方法 |
| ---- | ---- | -------- |
| 構造 | 所定の frontmatter、見出し、関連リンクを持つ | ルールブック、サンプル、lint を確認する |
| 一貫性 | 成果物カタログ、WBS、Schedule、RACI と矛盾しない | 関連文書を横断確認する |
| 利用可能性 | 人間と AI Agent が参照し、次の作業に使える粒度で記述する | サンプル、指示テンプレート、生成結果で確認する |
| 公開適性 | 個人情報、非公開情報、機密情報を含めない | 公開前レビューで確認する |

### 3.2. リスク管理

- リスクは、プロジェクト登録簿で識別し、必要に応じて [pm-risk-register.md](../controls/generated/pm-risk-register.md) に反映する。
- 重大リスクは、影響、発生可能性、対応方針、責任ロール、期限を明示する。
- 次のいずれかに該当する場合、リスクとして記録する。

| 条件 | 例 |
| ---- | -- |
| 成果物の完成や公開判断に影響する | 主要ルールブックの未整備、文書構成の不整合 |
| 継続運用に影響する | 更新負荷の増大、個人運用への過度な依存 |
| 利用者や貢献者に影響する | README や導入手順の不足、ライセンス未確定 |
| AI Agent の作業品質に影響する | 指示テンプレートの不足、入力文書の曖昧さ |

### 3.3. 課題管理

- 発生した課題は、プロジェクト登録簿で管理し、必要に応じて [pm-issue-log.md](../controls/generated/pm-issue-log.md) に反映する。
- 課題には、状態、優先度、担当ロール、期限、対応方針を記録する。
- 課題がスコープ、スケジュール、品質、公開判断に影響する場合は、`PO` が優先順位と対応方針を決定する。

### 3.4. 変更管理

- スコープ、主要成果物、公開方針、ライセンス方針、GO / Not GO 判断に影響する変更は、変更要求として扱う。
- 変更要求は、プロジェクト登録簿で起票し、必要に応じて [pm-change-request-log.md](../controls/generated/pm-change-request-log.md) に反映する。
- 軽微な表記修正、リンク修正、誤字修正は通常の文書更新として扱う。ただし、意味や責任分担が変わる場合は変更要求に含める。

## 4. 体制と意思決定

### 4.1. 採用ロール

採用ロールと未採用ロールは [pm-organization.md](../020-organization/pm-organization.md) を正とする。

| Role code | 主な責務 | 本計画での扱い |
| --------- | -------- | -------------- |
| `PO` | 最終判断、優先順位、公開方針、変更判断 | 管理責務と承認責任を集約する |
| `BA` | 利用者視点、要件、受入条件 | 利用可能性と導入しやすさを確認する |
| `ARC` | 文書体系、構成、技術制約 | 構成、命名、派生関係を確認する |
| `QE` | 品質基準、レビュー、整合性 | ルール適合、矛盾、公開適性を確認する |

### 4.2. 意思決定

| 判断対象 | 判断者 | 相談先 | 証跡 |
| -------- | ------ | ------ | ---- |
| プロジェクト目的、スコープ、優先順位 | `PO` | `BA`, `ARC`, `QE` | プロジェクト登録簿、決定記録、関連文書 |
| 成果物追加・削除・統合 | `PO` | `ARC`, `QE` | 成果物カタログ、WBS、RACI |
| 品質基準、レビュー基準 | `PO` | `QE` | 品質管理計画、レビュー記録 |
| 公開可否、ライセンス方針 | `PO` | `ARC`, `QE` | 決定記録、README、LICENSE |
| AI Agent の出力採否 | `PO` | 必要に応じて各ロール | 変更差分、レビュー結果 |

### 4.3. RACI

成果物別、プロセス別の責任分担は [pm-raci.md](../020-organization/pm-raci.md) を正とする。

本書では、`pm-plan` の責任分担を次のとおり扱う。

| 成果物 | PO | BA | ARC | QE |
| ------ | -- | -- | --- | -- |
| `pm-plan` | A/R | C | C | C |

## 5. レポーティングと見直し

### 5.1. レポーティング

コミュニケーションと報告の詳細は [pm-communication-plan.md](pm-communication-plan.md) で定義する。

本計画の段階では、次の最小運用を採用する。

| 管理領域 | 指標 | 報告頻度 | 判断者 | 閾値超過時対応 |
| -------- | ---- | -------- | ------ | -------------- |
| スコープ | 未決事項、追加成果物、変更要求 | 作業節目ごと | `PO` | 優先順位を見直し、必要に応じて WBS / Schedule を更新する |
| スケジュール | 期限超過、依存関係の停滞 | 作業節目ごと | `PO` | マイルストーン影響を確認し、タスク順序を見直す |
| 品質 | lint 結果、レビュー指摘、文書間不整合 | 主要成果物更新時 | `PO` | 修正方針を決め、該当文書と生成物を更新する |
| リスク・課題 | open 件数、高優先度項目、期限超過 | 作業節目ごと | `PO` | 対応期限、担当ロール、回避策を見直す |
| 公開準備 | README、LICENSE、導入手順、公開対象 | 初回公開前 | `PO` | 公開判断を保留し、必要な成果物を追加する |

### 5.2. 見直し条件

次のいずれかに該当する場合、本書を見直す。

| 更新トリガー | 見直し内容 |
| ------------ | ---------- |
| プロジェクトスコープまたは成功基準を変更した | 管理方針、品質基準、関連ドキュメントを更新する |
| 成果物カタログ、WBS、Schedule を大きく変更した | 管理単位、報告単位、RACI への影響を確認する |
| 採用ロールまたはメンバー構成を変更した | 意思決定、責任分担、エスカレーションを更新する |
| リスク、課題、変更要求の管理方法を変更した | 管理台帳、生成ビュー、報告方法を更新する |
| 初回公開または外部参加者の受け入れを開始した | 公開運用、問い合わせ対応、貢献導線を追加する |

### 5.3. GO / Not GO 判断

本格実行開始の GO / Not GO 判断は [prj-charter.md](../../020-project-definition/prj-charter.md) に従う。

判断時には、少なくとも次を確認する。

- プロジェクトスコープと成功基準が定義されていること。
- 成果物カタログ、WBS、Schedule の整合性が確認されていること。
- 管理計画、組織定義、RACI、品質管理、コミュニケーション方針が矛盾していないこと。
- 主要なリスク、課題、未決事項が識別され、対応方針があること。
- 公開対象に個人情報、非公開情報、機密情報が含まれていないこと。

## 7. 継続確認事項

| 論点 | 現状 | 対応方針 | 判断者 |
| ---- | ---- | -------- | ------ |
| プロジェクト定義文書の更新 | スコープ、成功基準、前提・制約・依存関係は draft として作成済み | 内容を更新した場合は、本書のスコープ管理、GO / Not GO 判断、関連ドキュメントへ反映する | `PO` |
| 個別管理計画の更新 | 品質管理計画、コミュニケーション計画は draft として作成済み | 管理方針を変更した場合は、本書の品質・報告・見直し条件へ反映する | `PO`, `QE` |
| 初回公開判断 | _UNDECIDED_ | README、LICENSE、公開範囲、個人情報・機密情報確認後に判断する | `PO` |
